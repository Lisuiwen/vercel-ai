# 将 AI SDK 6.x 迁移到 7.0

## 建议的迁移流程

1. 备份项目。若使用版本控制系统，请确保历史版本均已提交。
1. 升级到 AI SDK 7.0。
1. 遵循下方的破坏性变更指南。
1. 验证项目按预期工作。
1. 提交你的更改。

示例升级命令：

```
pnpm install ai@latest @ai-sdk/react@latest @ai-sdk/openai@latest @ai-sdk/otel@latest
```

## 所有包

### 仅 ESM — 已移除 CommonJS 支持

所有 AI SDK 包现在仅为 ESM。不再支持 `require()`。

若项目使用 CommonJS（`require()`），请改用 ESM `import` 语法：

```tsx filename="Before (CommonJS)"
const { generateText } = require('ai');
const { openai } = require('@ai-sdk/openai');
```

```tsx filename="After (ESM)"
import { generateText } from 'ai';
import { openai } from '@ai-sdk/openai';
```

若 `package.json` 尚未包含 `"type": "module"`，请添加该项或将文件改为 `.mjs` 扩展名。

## AI SDK Core

### Provider 管理：移除已弃用的 `experimental_customProvider`

AI SDK 7 已移除已弃用的 `experimental_customProvider` 导出。请改用 `customProvider`。

```tsx filename="AI SDK 6"
import { experimental_customProvider } from 'ai';

export const myProvider = experimental_customProvider({
  languageModels: {
    // ...
  },
});
```

```tsx filename="AI SDK 7"
import { customProvider } from 'ai';

export const myProvider = customProvider({
  languageModels: {
    // ...
  },
});
```

这只是 import 与符号重命名。`customProvider` 的选项与行为不变。

### 提示指令：`system` 重命名为 `instructions`

用于系统指令的顶层提示选项已由 `system` 重命名为 `instructions`。

```tsx filename="AI SDK 6"
const result = await generateText({
  model: yourModel,
  system: 'You are a helpful assistant.',
  prompt: 'Hello!',
});
```

```tsx filename="AI SDK 7"
const result = await generateText({
  model: yourModel,
  instructions: 'You are a helpful assistant.',
  prompt: 'Hello!',
});
```

适用于接受 `prompt` 或 `messages` 的 AI SDK 函数，包括
`generateText`、`streamText`、`generateObject`、`streamObject` 与 `streamUI`。

同样的重命名也适用于 `generateText` 与 `streamText` 的 `prepareStep` 返回值，以及传入 `experimental_repairToolCall` 的选项：

```tsx filename="AI SDK 6"
const result = streamText({
  model: yourModel,
  prompt: 'Hello!',
  prepareStep: () => ({
    system: 'Use concise answers for this step.',
  }),
});
```

```tsx filename="AI SDK 7"
const result = streamText({
  model: yourModel,
  prompt: 'Hello!',
  prepareStep: () => ({
    instructions: 'Use concise answers for this step.',
  }),
});
```

`system` 选项仍可作为已弃用的回退项接受。当同时提供 `instructions` 与 `system` 时，`instructions` 优先。

#### `prepareStep` 指令会延续到后续步骤

在 AI SDK 7 中，从 `prepareStep` 返回的指令会继续用于后续步骤，
直到 `prepareStep` 再次返回不同的 `instructions` 或 `system` 覆盖。
这与 `prepareStep` 返回的 `messages` 如何延续一致。

在 AI SDK 6 中，`prepareStep` 的指令覆盖仅作用于当前步骤。
后续步骤会回退到顶层 `system` 指令，除非再次返回覆盖。

若你的 `prepareStep` 逻辑依赖「仅一步」的指令覆盖，
请为每一步显式返回所需指令：

```tsx filename="AI SDK 7"
const result = streamText({
  model: yourModel,
  instructions: 'Use the default behavior.',
  prompt: 'Hello!',
  prepareStep: ({ stepNumber, initialInstructions }) => ({
    instructions:
      stepNumber === 0
        ? 'Use special instructions for the first step.'
        : initialInstructions,
  }),
});
```

`prepareStep` 回调会收到 `instructions`（当前步骤的指令状态）与 `initialInstructions`（最初调用时的顶层指令值）。

若工具调用修复函数会将当前系统指令转发给另一次模型调用，
请读取并传递 `instructions`，而不是 `system`：

```tsx filename="AI SDK 6"
const result = await generateText({
  model: yourModel,
  tools,
  prompt: 'Hello!',
  experimental_repairToolCall: async ({ system, messages }) => {
    return repairWithModel({ system, messages });
  },
});
```

```tsx filename="AI SDK 7"
const result = await generateText({
  model: yourModel,
  tools,
  prompt: 'Hello!',
  experimental_repairToolCall: async ({ instructions, messages }) => {
    return repairWithModel({ instructions, messages });
  },
});
```

`generateText`、`streamText` 以及 agents 的生命周期回调事件也同样使用 `instructions` 代替 `system`：

```tsx filename="AI SDK 6"
const result = await generateText({
  model: yourModel,
  prompt: 'Hello!',
  experimental_onStart: ({ system }) => {
    console.log(system);
  },
});
```

```tsx filename="AI SDK 7"
const result = await generateText({
  model: yourModel,
  prompt: 'Hello!',
  experimental_onStart: ({ instructions }) => {
    console.log(instructions);
  },
});
```

### 提示消息：默认拒绝 `prompt` 或 `messages` 中的系统消息

AI SDK 7 默认会拒绝出现在 `prompt` 或 `messages` 字段中的系统消息。
系统指令通常应通过顶层 `instructions` 选项传入。

这可能破坏包含 `{ role: 'system' }` 的旧持久化会话或自定义提示数组：

```tsx filename="AI SDK 6"
const result = await generateText({
  model: yourModel,
  messages: [
    { role: 'system', content: 'You are a helpful assistant.' },
    { role: 'user', content: 'Hello!' },
  ],
});
```

若可能，请将系统指令移到 `instructions` 选项：

```tsx filename="AI SDK 7"
const result = await generateText({
  model: yourModel,
  instructions: 'You are a helpful assistant.',
  messages: [{ role: 'user', content: 'Hello!' }],
});
```

若需要保留已包含系统消息的既有会话历史，可通过 `allowSystemInMessages: true` 选择恢复旧行为：

<Note type="warning">
  仅在消息可信时使用 `allowSystemInMessages`。若用户可以提交或编辑这些消息，
  他们可能注入系统消息来覆盖或设置系统提示。大多数情况下，系统指令应仅由可信的服务端代码通过 `instructions` 属性设置。
</Note>

```tsx filename="AI SDK 7"
const result = await generateText({
  model: yourModel,
  allowSystemInMessages: true,
  messages: persistedMessages,
});
```

适用于接受 `prompt` 或 `messages` 的 AI SDK 函数，包括 `generateText`、`streamText`、`generateObject`、`streamObject` 与 `streamUI`。

### 从 `LanguageModelUsage` 移除 `cachedInputTokens` 与 `reasoningTokens`

已弃用的顶层字段 `cachedInputTokens` 与 `reasoningTokens` 已从 `LanguageModelUsage` 移除。

请改用 `inputTokenDetails.cacheReadTokens` 与 `outputTokenDetails.reasoningTokens`：

```tsx filename="AI SDK 6"
const result = await generateText({
  model: yourModel,
  prompt: 'Hello!',
});

console.log(result.usage.cachedInputTokens);
console.log(result.usage.reasoningTokens);
```

```tsx filename="AI SDK 7"
const result = await generateText({
  model: yourModel,
  prompt: 'Hello!',
});

console.log(result.usage.inputTokenDetails.cacheReadTokens);
console.log(result.usage.outputTokenDetails.reasoningTokens);
```

### Telemetry：`OpenTelemetry` 迁至 `@ai-sdk/otel` — 需要显式注册

OpenTelemetry span 收集不再内置于 `ai` 包。要继续接收 OpenTelemetry trace，你必须安装新的 `@ai-sdk/otel` 包并在全局注册 `OpenTelemetry` 实例。

```bash
pnpm install @ai-sdk/otel
```

此前，当启用 `experimental_telemetry` 时会自动发出 OpenTelemetry span：

```tsx filename="AI SDK 6"
import { generateText } from 'ai';

const result = await generateText({
  model: yourModel,
  prompt: 'Hello',
  experimental_telemetry: { isEnabled: true },
});
```

现在，你必须安装 `@ai-sdk/otel` 并在应用启动时注册一次 `OpenTelemetry` 实例。对于 Next.js，请将其放在 `instrumentation.ts` 中，与 OpenTelemetry provider 设置并列：

```tsx filename="instrumentation"
import { registerTelemetry } from 'ai';
import { OpenTelemetry } from '@ai-sdk/otel';

registerTelemetry(new OpenTelemetry());

// ... your OpenTelemetry provider setup (e.g. registerOTel, NodeTracerProvider)
```

对于不使用 Next.js 的 Node.js 应用，请在入口文件顶层注册集成。

适用于所有接受 `experimental_telemetry` 的 AI SDK 函数，包括 `generateText`、`streamText`、`ToolLoopAgent`、`embed`、`embedMany` 与 `rerank`。

### Telemetry：注册集成后默认启用

在 AI SDK 6 中，telemetry 需要手动开启——你必须在每次调用时设置 `experimental_telemetry: { isEnabled: true }` 才会发出事件。
在 AI SDK 7 中，telemetry 改为默认开启后的可关闭模式：一旦你注册了 telemetry 集成（例如 `OpenTelemetry` 或 `DevToolsTelemetry`），所有 AI SDK 调用默认都会发出 telemetry 事件。

```tsx filename="AI SDK 6"
import { generateText } from 'ai';

const result = await generateText({
  model: yourModel,
  prompt: 'Hello',
  experimental_telemetry: { isEnabled: true },
});
```

```tsx filename="AI SDK 7"
import { generateText } from 'ai';
import { registerTelemetry } from 'ai';
import { OpenTelemetry } from '@ai-sdk/otel';

registerTelemetry(new OpenTelemetry());

const result = await generateText({
  model: yourModel,
  prompt: 'Hello',
});
```

你可以安全地从所有调用中移除 `experimental_telemetry: { isEnabled: true }`。若已在传入 `functionId`、`integrations` 等字段，请保留它们——只有 `isEnabled: true` 是多余的：

```tsx filename="AI SDK 6"
const result = await generateText({
  model: yourModel,
  prompt: 'Hello',
  experimental_telemetry: {
    isEnabled: true,
    functionId: 'my-awesome-function',
  },
});
```

```tsx filename="AI SDK 7"
const result = await generateText({
  model: yourModel,
  prompt: 'Hello',
  experimental_telemetry: {
    functionId: 'my-awesome-function',
  },
});
```

若要在单次调用中关闭 telemetry，请设置 `isEnabled: false`：

```tsx
const result = await generateText({
  model: yourModel,
  prompt: 'Hello',
  experimental_telemetry: { isEnabled: false },
});
```

若要全局禁用 telemetry，请不要注册任何 telemetry 集成。

适用于所有接受 `experimental_telemetry` 的 AI SDK 函数，包括 `generateText`、`streamText`、`ToolLoopAgent`、`embed`、`embedMany` 与 `rerank`。

### Telemetry：已从 `experimental_telemetry` 移除 `tracer` 属性

`experimental_telemetry` 上的 `tracer` 属性已移除。若你此前传入自定义 OpenTelemetry `Tracer`，请改传给 `OpenTelemetry` 构造函数：

```tsx filename="AI SDK 6"
import { generateText } from 'ai';
import { trace } from '@opentelemetry/api';

const result = await generateText({
  model: yourModel,
  prompt: 'Hello',
  experimental_telemetry: {
    isEnabled: true,
    tracer: trace.getTracer('my-app'),
  },
});
```

```tsx filename="AI SDK 7"
import { registerTelemetry } from 'ai';
import { OpenTelemetry } from '@ai-sdk/otel';
import { trace } from '@opentelemetry/api';

registerTelemetry(
  new OpenTelemetry({
    tracer: trace.getTracer('my-app'),
  }),
);
```

适用于所有接受 `experimental_telemetry` 的 AI SDK 函数，包括 `streamText`、`generateObject`、`streamObject`、`embed` 与 `embedMany`。

若你未传入自定义 `tracer`（依赖默认全局 tracer），则无需改动——`OpenTelemetry` 默认全局注册，且在未提供自定义 tracer 时使用 `trace.getTracer('ai')`。

### Telemetry：`experimental_telemetry` 重命名为 `telemetry`

telemetry 选项已脱离实验状态，并改名为 `telemetry`。旧名称 `experimental_telemetry` 在 AI SDK 7 中仍可作为已弃用别名使用，并将在未来的主版本中移除。

```tsx filename="AI SDK 6"
import { generateText } from 'ai';

const result = await generateText({
  model: yourModel,
  prompt: 'Hello',
  experimental_telemetry: {
    functionId: 'story-agent',
  },
});
```

```tsx filename="AI SDK 7"
import { generateText } from 'ai';

const result = await generateText({
  model: yourModel,
  prompt: 'Hello',
  telemetry: {
    functionId: 'story-agent',
  },
});
```

适用于所有接受 telemetry 配置的 AI SDK 函数与 agents，包括 `generateText`、`streamText`、`generateObject`、`streamObject`、`embed`、`embedMany`、`rerank`、`ToolLoopAgent` 与 `WorkflowAgent`。

除重命名外行为不变。你可以渐进迁移——在弃用窗口期内，同一代码库中混用 `telemetry` 与 `experimental_telemetry` 是受支持的。

### 工具执行回调：`experimental_onToolCallStart` 与 `experimental_onToolCallFinish` 重命名

`generateText` 与 `streamText` 的工具执行回调选项已重命名：

- `experimental_onToolCallStart` 现为 `onToolExecutionStart`
- `experimental_onToolCallFinish` 现为 `onToolExecutionEnd`

旧名称在 AI SDK 7 中仍可作为已弃用别名使用，并将在未来的主版本中移除。它们仅在新回调名未提供时作为回退。

```tsx filename="AI SDK 6"
const result = await generateText({
  model: yourModel,
  tools,
  prompt: 'Hello',
  experimental_onToolCallStart(event) {
    console.log('Tool starting:', event.toolCall.toolName);
  },
  experimental_onToolCallFinish(event) {
    console.log('Tool finished:', event.toolCall.toolName);
  },
});
```

```tsx filename="AI SDK 7"
const result = await generateText({
  model: yourModel,
  tools,
  prompt: 'Hello',
  onToolExecutionStart(event) {
    console.log('Tool starting:', event.toolCall.toolName);
  },
  onToolExecutionEnd(event) {
    console.log('Tool finished:', event.toolCall.toolName);
  },
});
```

### Telemetry Rerank 回调：`onRerankFinish` 重命名为 `onRerankEnd`

单次 rerank 模型调用的 telemetry 集成回调已由 `onRerankFinish` 重命名为 `onRerankEnd`。

这同样适用于发布在 `AI_SDK_TELEMETRY_DIAGNOSTIC_CHANNEL` 上的 `type` 字段：diagnostic-channel 订阅者现在会收到 `onRerankEnd` 而非 `onRerankFinish`。

请将 telemetry 集成与 diagnostic-channel 订阅者更新为使用 `onRerankEnd`。

```tsx filename="AI SDK 6"
import type { Telemetry } from 'ai';

const telemetry: Telemetry = {
  onRerankFinish(event) {
    console.log('Rerank finished:', event.ranking.length);
  },
};
```

```tsx filename="AI SDK 7"
import type { Telemetry } from 'ai';

const telemetry: Telemetry = {
  onRerankEnd(event) {
    console.log('Rerank ended:', event.ranking.length);
  },
};
```

### Telemetry Embed 回调：`onEmbedFinish` 重命名为 `onEmbedEnd`

单次 embedding 模型调用的 telemetry 集成回调已由 `onEmbedFinish` 重命名为 `onEmbedEnd`。

这同样适用于发布在 `AI_SDK_TELEMETRY_DIAGNOSTIC_CHANNEL` 上的 `type` 字段：diagnostic-channel 订阅者现在会收到 `onEmbedEnd` 而非 `onEmbedFinish`。

请将 telemetry 集成与 diagnostic-channel 订阅者更新为使用 `onEmbedEnd`。

```tsx filename="AI SDK 6"
import type { Telemetry } from 'ai';

const telemetry: Telemetry = {

    console.log('Embedding finished:', event.embeddings.length);
  },
};
```

```tsx filename="AI SDK 7"
import type { Telemetry } from 'ai';

const telemetry: Telemetry = {
  onEmbedEnd(event) {
    console.log('Embedding ended:', event.embeddings.length);
  },
};
```

已完成 `embed` 与 `embedMany` 调用的逐调用回调选项已由 `experimental_onFinish` 重命名为 `experimental_onEnd`。

```tsx filename="AI SDK 6"
const result = await embed({
```

### Rerank 回调：`experimental_onFinish` 重命名为 `experimental_onEnd`

已完成 `rerank` 调用的逐调用回调选项已由 `experimental_onFinish` 重命名为 `experimental_onEnd`。

```tsx filename="AI SDK 6"
const result = await rerank({
```

```tsx filename="AI SDK 7"
const result = await rerank({
```

### 流式：`includeRawChunks` 迁至 `include.rawChunks`

AI SDK 7 中 `streamText` 的顶层 `includeRawChunks` 选项已弃用。请将其移到 `include` 选项对象中，字段名为 `rawChunks`。

```tsx filename="AI SDK 6"
const result = streamText({
  model: yourModel,
  prompt: 'Hello',
  includeRawChunks: true,
});
```

```tsx filename="AI SDK 7"
const result = streamText({
  model: yourModel,
  prompt: 'Hello',
  include: {
    rawChunks: true,
  },
});
```

顶层弃用选项在 AI SDK 7 中仍可用，因此你可以渐进迁移。

### 将 `experimental_include` 重命名为 `include`

面向 `generateText`、`streamText` 与 `ToolLoopAgent` 的 `experimental_include` 选项现已稳定，并重命名为 `include`。

```tsx filename="AI SDK 6"
const result = await generateText({
  model: yourModel,
  prompt: 'Hello',
  experimental_include: {
    requestBody: false,
  },
});
```

```tsx filename="AI SDK 7"
const result = await generateText({
  model: yourModel,
  prompt: 'Hello',
  include: {
    requestBody: false,
  },
});
```

已弃用的 `experimental_include` 名称仍可向后兼容使用。若同时提供 `include` 与 `experimental_include`，则 `include` 优先。

### 默认不包含请求与响应体

在 AI SDK 7 中，`generateText` 与 `streamText` 默认不再在 step 结果中包含请求体；`generateText` 默认也不再包含响应体。
这可降低内存占用，尤其在提示词或提供商响应包含图片或文件等大载荷时。

若应用读取 `result.request.body`、`step.request.body`、`result.response.body` 或 `step.response.body`，请通过 `include` 主动开启。

```tsx filename="AI SDK 7"
const result = await generateText({
  model: yourModel,
  prompt: 'Hello',
  include: {
    requestBody: true,
    responseBody: true,
  },
});
```

对 `streamText` 而言，仅支持 `requestBody`：

```tsx filename="AI SDK 7"
const result = streamText({
  model: yourModel,
  prompt: 'Hello',
  include: {
    requestBody: true,
  },
});
```

### Step 响应消息不再累积

在 AI SDK 7 中，每个 `StepResult` 上的 `step.response.messages` **仅包含该步骤产生的**响应消息。

在 AI SDK 6 中，`step.response.messages` 会累积此前所有步骤的响应消息。若你的应用从中途或最终步骤读取 `step.response.messages`，并期望拿到完整的助手/工具消息历史，
请改用顶层 `result.responseMessages`。

```tsx filename="AI SDK 7"
const result = await generateText({
  model: yourModel,
  prompt: 'Use tools if needed',
  tools,
  stopWhen: isStepCount(5),
});

// Accumulated response messages from all steps:
const responseMessages = result.responseMessages;

// Response messages produced by each individual step:
const stepMessages = result.steps.map(step => step.response.messages);
```

若你确实需要从 step 结果重建响应消息，
可将各步骤消息扁平化：

```tsx filename="AI SDK 7"
const responseMessages = result.steps.flatMap(step => step.response.messages);
```

当你需要完整响应消息历史时，优先使用 `result.responseMessages`。
它还会包含第一个模型步骤之前产生的响应消息，例如输入消息里已批准工具调用对应的工具结果。

### Context：`experimental_context` 变为工具 `context`，共享运行时数据迁至 `runtimeContext`

在 AI SDK 7 中，此前以 `experimental_context` 暴露的工具回调选项已重命名为 `context`，并已稳定。

更重要的行为变化是 AI SDK 现在区分两类数据：

- 工具专属的 `context`，通过 `toolsContext` 传入工具回调
- 共享的生成/agent 运行时数据，通过 `runtimeContext` 流转

在 AI SDK 6 中，工具常常从同一个通用运行时上下文对象读取数据。在 AI SDK 7 中，每个工具拥有各自作用域的 `context`，你需要按工具名将值传入 `toolsContext`。

`tool()` 会根据 `contextSchema` 推断工具的 `context` 类型，因此每个工具只能看到为该工具声明的字段。

在 AI SDK 6 中，工具代码经常访问 `experimental_context` 并需要手动断言：

```tsx filename="AI SDK 6"
const weather = tool({
  inputSchema: z.object({
    location: z.string(),
  }),
  execute: async ({ location }, { experimental_context }) => {
    const { weatherApiKey } = experimental_context as {
      weatherApiKey: string;
    };

    return getWeather(location, weatherApiKey);
  },
});
```

在 AI SDK 7 中，将 `experimental_context` 重命名为 `context`，用 `contextSchema` 声明工具专属上下文，并通过 `toolsContext` 传入每个工具的值。随后 `execute` 回调、审批回调以及工具输入生命周期钩子会自动收到类型化的 `context`：

```tsx filename="AI SDK 7"
const weather = tool({
  inputSchema: z.object({
    location: z.string(),
  }),
  contextSchema: z.object({
    apiKey: z.string(),
  }),
  execute: async ({ location }, { context: { apiKey } }) => {
    return getWeather(location, apiKey);
  },
});

const result = await generateText({
  model: yourModel,
  tools: { weather },
  runtimeContext: {
    requestId: 'req-123',
  },
  toolsContext: {
    weather: {
      apiKey: process.env.WEATHER_API_KEY!,
    },
  },
  prepareStep: async ({ runtimeContext, toolsContext }) => {
    console.log(runtimeContext.requestId);
    console.log(toolsContext.weather.apiKey);

    return {};
  },
});
```

对需要在 `prepareStep`、事件与 step 结果中可见的共享生成或 agent 状态，请使用 `runtimeContext`。工具回调不再从该共享对象读取数据。工具的 `context` 现在来自 `toolsContext` 中对应工具的条目，因此只会包含该工具在 `contextSchema` 中声明的字段。

可能的破坏性影响：

- 若仍在工具回调中解构 `experimental_context`，请改名为 `context`。
- 若此前把同一个共享运行时对象既用于编排又用于工具，请拆分为 `runtimeContext` 与 `toolsContext`。
- 将工具专属值放到匹配工具名下的 `toolsContext` 中。
- 将 `generateText`、`streamText`、`ToolLoopAgent` 中共享的顶层 `context` 用法改名为 `runtimeContext`。
- 将 `prepareStep` 中的 `context` 改名为 `runtimeContext`。
- 若工具回调访问了未在其 `contextSchema` 中声明的字段，TypeScript 现在会报错。请把这些字段加入该工具的 schema。
- 若有辅助类型或包装器假定 `experimental_context` 或 `context` 为 `unknown`，请更新为接受泛型 `CONTEXT`。
- 若至少一个工具声明了 `contextSchema`，则 `toolsContext` 变为必填，且仅包含实际声明上下文数据的工具。
- 若你依赖每个工具回调都能看到同一个完整运行时对象，请将共享步骤数据搬到 `runtimeContext`，或在每个工具的 `toolsContext` 条目中显式提供所需的按工具数据。

### 工具结果内容：从 `image-*` 与 `file-*` 变体迁移到 `file`

面向 `toModelOutput` 结果的所有 `image-*` 与旧版 `file-*` 内容部件类型在 AI SDK 7 中已弃用，推荐使用单一规范的 `file` 变体，其形状与顶层 `FilePart` 对齐。运行时会自动迁移，因此既有工具输出无需改代码仍可工作；但仍建议更新到新形状。

新形状带有带标签的 `data` 判别联合，并且始终包含 `mediaType`：

- `{ type: 'file-data', data, mediaType, filename? }` → `{ type: 'file', mediaType, filename, data: { type: 'data', data } }`
- `{ type: 'file-url', url, mediaType }` → `{ type: 'file', mediaType, data: { type: 'url', url: new URL(url) } }`
- `{ type: 'file-reference', providerReference }` → `{ type: 'file', mediaType, data: { type: 'reference', reference: providerReference } }`

图片只是带有图片媒体类型的文件，因此 `image-*` 别名会合并到同一形状——请改用 `mediaType: 'image'`（或更具体的 `image/*` 子类型）：

- `{ type: 'image-data', data, mediaType }` → `{ type: 'file', mediaType, data: { type: 'data', data } }`
- `{ type: 'image-url', url }` → `{ type: 'file', mediaType: 'image', data: { type: 'url', url: new URL(url) } }`
- `{ type: 'image-file-reference', providerReference }` → `{ type: 'file', mediaType: 'image', data: { type: 'reference', reference: providerReference } }`

`-id` 内容部件类型（`file-id` 与 `image-file-id`）通常也已弃用，建议使用携带完整 `ProviderReference`（提供商到文件 ID 的映射，例如 `{ openai: 'file_123', anthropic: 'file_abc' }`）的 `reference` data 形状，从而让同一逻辑文件可跨提供商复用。单一 ID 的 `-id` 变体要求运行时知道 ID 属于哪个提供商；显式的 reference 形状消除了这种不确定性：

- `{ type: 'file-id', fileId }` → `{ type: 'file', mediaType, data: { type: 'reference', reference: { [provider]: fileId } } }`
- `{ type: 'image-file-id', fileId }` → `{ type: 'file', mediaType: 'image', data: { type: 'reference', reference: { [provider]: fileId } } }`

新版 `file` 变体上的 `mediaType` 既可接受完整 IANA 类型（例如 `'image/png'`），也可仅接受顶层分段（例如 `'image'`、`'audio'`、`'video'`、`'text'`）。仅提供顶层分段时，会在可行情况下从内联字节自动检测子类型。请更新解析/校验与判别联合，以处理这一单一规范的 `file` 变体。


### 消息部件：处理新的 `reasoning-file` 内容类型

部分模型现在可以将文件作为推理轨迹的一部分返回，并与常规输出文件区分开。这些此前会使用 `file` 类型的文件，现在属于独立的 `reasoning-file` 类型。

请更新所有穷尽的部件处理逻辑（TypeScript 联合类型、`switch`、运行时校验器、渲染器、序列化器）以支持 `type: 'reasoning-file'`。
