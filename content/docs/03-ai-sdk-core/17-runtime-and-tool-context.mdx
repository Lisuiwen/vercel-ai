# 运行时与 Tool 上下文

Context 让你在不把状态写入提示的情况下，将服务端状态传入生成或 agent 循环。
AI SDK 将共享的运行时状态与按 tool 的执行状态分开，使 agent 能跟踪整体进展，而 tool 只接收其所需的数据。

可将租户信息、功能开关、会话数据、请求 ID、API 凭证等应用状态作为 context 传入，以影响执行行为。

## 上下文类型

| 概念                           | 定义位置                                                | 读取位置                                                                               | 适用场景                                                    |
| --------------------------------- | ------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------- | ------------------------------------------------------------- |
| `runtimeContext`                  | `generateText`、`streamText` 或 `ToolLoopAgent` 调用             | `prepareStep`、生命周期回调、step 结果、遥测                                 | 整次生成或 agent 循环的共享状态                              |
| `toolsContext`                    | `generateText`、`streamText` 或 `ToolLoopAgent` 调用             | `prepareStep`、审批回调、tool context 解析、tool 描述函数      | 按 tool 名称映射的 context 值                           |
| tool `context`                    | 各 tool 在 `toolsContext` 中的条目，由 `contextSchema` 校验 | tool 描述函数、tool `execute`、`needsApproval`、tool 输入生命周期回调 | 单个 tool 所需的数据                                     |
| `toolContext`                     | 由单个 tool 的 context 派生                                    | tool 审批回调与 tool 执行事件                                               | 单个 tool context 在事件/回调中的名称                               |
| `telemetry.includeRuntimeContext` | 生成或 agent 调用                                       | 遥测过滤                                                                             | 要纳入遥测的顶层 `runtimeContext` 属性 |
| `telemetry.includeToolsContext`   | 生成或 agent 调用                                       | 遥测过滤                                                                             | 要纳入遥测的顶层 tool context 属性     |

在 agent 中，`runtimeContext` 是 agent 的共享运行时状态，在循环中流动，并可在模型调用之间的 `prepareStep` 中读取或更新。与特定 tool 相关的数据保留在 `toolsContext` 中，每个 tool 只接收经过校验的自有 `context`。

```txt
generateText / streamText / ToolLoopAgent
  -> runtimeContext
       -> prepareStep, lifecycle callbacks, step results
       -> telemetry, filtered by telemetry.includeRuntimeContext
  -> toolsContext
       -> prepareStep
       -> one tool's context / toolContext
            -> execute, approval callbacks, tool events
            -> telemetry, filtered by telemetry.includeToolsContext
```

## 运行时上下文

在需要为整次生成或 agent 循环提供共享状态时，传入 `runtimeContext`。
它不会自动加入模型提示，可用于配置 step 准备、跟踪服务端状态或关联生命周期事件。

```ts
const result = await generateText({
  model: __MODEL__,
  prompt: 'Help the user plan their trip.',
  runtimeContext: {
    tenantId: 'tenant_123',
    plan: 'enterprise',
    requestId: 'req_abc',
  },
  prepareStep: async ({ runtimeContext }) => {
    if (runtimeContext.plan === 'enterprise') {
      return { temperature: 0.2 };
    }

    return {};
  },
});
```

`prepareStep` 可返回新的 `runtimeContext`。新值会影响当前 step 及之后所有 step，适合在循环各轮之间更新 agent 状态。

## Tool 上下文

使用 `toolsContext` 存放属于特定 tool 的值。每个 tool 通过 `contextSchema` 声明期望的 context；匹配的 `toolsContext` 条目经校验后作为 `context` 传入 tool。

```ts highlight="11-15,26-31"
import { generateText, tool } from 'ai';
import { z } from 'zod';

const weatherTool = tool({
  description: 'Get the weather in a location',
  inputSchema: z.object({
    location: z.string(),
  }),
  contextSchema: z.object({
    weatherApiKey: z.string(),
    defaultUnit: z.enum(['celsius', 'fahrenheit']),
  }),
  execute: async ({ location }, { context }) => {
    return fetchWeather({
      location,
      apiKey: context.weatherApiKey,
      unit: context.defaultUnit,
    });
  },
});

const result = await generateText({
  model: __MODEL__,
  tools: { weather: weatherTool },
  toolsContext: {
    weather: {
      weatherApiKey: process.env.WEATHER_API_KEY!,
      defaultUnit: 'fahrenheit',
    },
  },
  prompt: 'What is the weather in San Francisco?',
});
```

当至少一个 tool 声明了 `contextSchema` 时，需要 context 的 tool 必须提供 `toolsContext`。每个 tool 只接收自己的 context，而非整个 `toolsContext` 映射。tool 描述函数在每次模型调用前也会收到相同类型的 `context`，因此描述可随当前 tool context 变化。

在 tool 内部应将 tool context 视为不可变。若需在 step 之间修改 tool context，可在 `prepareStep` 中检查上一步并返回更新后的 `toolsContext`。

## 遥测上下文过滤

Context 常包含对应用有用但不应发给遥测 provider 的值。使用 `telemetry.includeRuntimeContext` 选择要纳入遥测的顶层 runtime context 属性；使用 `telemetry.includeToolsContext` 按 tool 选择要纳入遥测的顶层 tool context 属性。

```ts highlight="36-45"
import { ToolLoopAgent, tool } from 'ai';
import { z } from 'zod';

const customerLookup = tool({
  description: 'Look up customer account details',
  inputSchema: z.object({
    customerId: z.string(),
  }),
  contextSchema: z.object({
    apiKey: z.string(),
    region: z.string(),
  }),
  execute: async ({ customerId }, { context }) => {
    return lookupCustomer({
      customerId,
      apiKey: context.apiKey,
      region: context.region,
    });
  },
});

const agent = new ToolLoopAgent({
  model: __MODEL__,
  tools: { customerLookup },
});

const result = await agent.generate({
  prompt: 'Check whether customer cust_123 is eligible for priority support.',
  runtimeContext: {
    requestId: 'req_abc',
    tenantId: 'tenant_123',
    userId: 'user_123',
  },
  telemetry: {
    includeRuntimeContext: {
      requestId: true,
    },
    includeToolsContext: {
      customerLookup: {
        region: true,
      },
    },
  },
  toolsContext: {
    customerLookup: {
      apiKey: process.env.CUSTOMER_API_KEY!,
      region: 'us',
    },
  },
});
```

本例中，遥测收到的 `runtimeContext` 仅含 `requestId`，`customerLookup` 的 context 仅含 `region`。

<Note>
  上下文过滤仅影响遥测集成（含 OpenTelemetry）。
  tool 执行、生命周期回调与返回结果仍接收完整 context 值。
</Note>

上下文遥测过滤为浅层过滤。对 `telemetry.includeRuntimeContext`，配置后仅发送标记为 `true` 的顶层属性；省略则不发送任何 runtime context 属性。对 `telemetry.includeToolsContext`，配置后仅发送各 tool 标记为 `true` 的顶层属性；省略则不发送任何 tool context 属性。

## 上下文可用位置

| 位置                          | `runtimeContext`                              | `toolsContext`                              | Tool `context` / `toolContext`                           |
| --------------------------------- | --------------------------------------------- | ------------------------------------------- | -------------------------------------------------------- |
| `prepareStep`                     | 读取与更新                               | 读取与更新                             | 不直接可用                                             |
| Tool 描述函数        | 不直接传入                           | 不直接传入                         | 读取单个 tool 的已校验 `context`                      |
| Tool `execute`                    | 不直接传入                           | 不直接传入                         | 读取单个 tool 的已校验 `context`                      |
| Tool 审批                     | 在通用与按 tool 回调中读取        | 在通用回调中读取                   | 在按 tool 回调中作为 `toolContext` 读取              |
| Tool 执行事件             | 不包含                                  | 不包含                                | 作为 `toolContext` 读取                                    |
| Step 结果与完成回调 | 读取最终或每步状态                  | 读取最终或每步状态                | 通过 `toolsContext` 中各 tool 条目可用 |
| 遥测                         | 由 `telemetry.includeRuntimeContext` 过滤 | 由 `telemetry.includeToolsContext` 过滤 | 由 `telemetry.includeToolsContext` 过滤              |

## 如何选择上下文

- 使用 `runtimeContext` 存放整次生成或 agent 循环的共享状态，如请求元数据、租户设置、功能开关或 agent 进度。
- 使用 `toolsContext` 与 `contextSchema` 存放特定 tool 所需的数据，如 API 密钥、作用域客户端、用户权限或 tool 默认设置。
- 使用提示消息存放模型需要推理或应在回答中提及的信息。
- 使用 `telemetry.includeRuntimeContext` 与 `telemetry.includeToolsContext` 减少遥测暴露范围，而非作为通用安全边界。
