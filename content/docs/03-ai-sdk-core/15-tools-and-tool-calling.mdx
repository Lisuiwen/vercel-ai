# Tool 调用

如 Foundations 所述，[tools](/docs/foundations/tools) 是模型可调用的对象，用于执行特定任务。
函数 tool 与动态 tool 包含以下核心要素：

- **`description`**：可选的 tool 描述，可影响何时选用该 tool。可为字符串，或根据 tool 的 context 与 experimental sandbox 派生描述的函数。
- **`inputSchema`**：定义输入参数的 [Zod schema](/docs/foundations/tools#schemas) 或 [JSON schema](/docs/reference/ai-sdk-core/json-schema)。schema 供 LLM 消费，也用于校验 LLM 的 tool 调用。
- **`execute`**：可选的异步函数，以 tool 调用的输入调用，产生 `RESULT` 类型（泛型）的值。可选是因为你可能将 tool 调用转发到客户端或队列，而非在同一进程执行。
- **`strict`**：_（可选，boolean）_ 在 provider 支持时启用严格 tool 调用

<Note className="mb-2">
  可使用 [`tool`](/docs/reference/ai-sdk-core/tool) 辅助函数推断 `execute` 参数的类型。
</Note>

`generateText` 与 `streamText` 的 `tools` 参数是以 tool 名称为键、tool 为值的对象：

```ts highlight="6-17"
import { z } from 'zod';
import { generateText, tool, isStepCount } from 'ai';
__PROVIDER_IMPORT__;

const result = await generateText({
  model: __MODEL__,
  tools: {
    weather: tool({
      description: 'Get the weather in a location',
      inputSchema: z.object({
        location: z.string().describe('The location to get the weather for'),
      }),
      execute: async ({ location }) => ({
        location,
        temperature: 72 + Math.floor(Math.random() * 21) - 10,
      }),
    }),
  },
  stopWhen: isStepCount(5),
  prompt: 'What is the weather in San Francisco?',
});
```

<Note>
  模型使用 tool 时称为「tool call」，tool 的输出称为「tool result」。
</Note>

Tool 调用不限于文本生成，
也可用于渲染用户界面（Generative UI）。

## 动态描述

Tool 描述可以是固定字符串或函数。当发给模型的描述应依赖当前 tool context 或 experimental sandbox（如租户、项目、环境或工作区）时使用函数。

描述函数在每一步将 tool 定义发给模型之前解析。它从 `toolsContext` 接收匹配的 tool `context`，以及已提供的当前 `experimental_sandbox`。若 `prepareStep` 更新了 `toolsContext` 或 `experimental_sandbox`，下一步会使用更新后的值。

```ts highlight="6-14,28-33"
import { generateText, tool } from 'ai';
import { z } from 'zod';

const shell = tool({
  contextSchema: z.object({
    projectName: z.string(),
  }),
  description: ({ context, experimental_sandbox }) =>
    [
      `Run shell commands for the ${context.projectName} project.`,
      experimental_sandbox != null
        ? `Sandbox: ${experimental_sandbox.description}`
        : undefined,
    ]
      .filter(Boolean)
      .join('\n'),
  inputSchema: z.object({
    command: z.string(),
  }),
  execute: async ({ command }, { experimental_sandbox }) => {
    if (!experimental_sandbox) {
      throw new Error('Experimental sandbox is not available');
    }

    return experimental_sandbox.runCommand({ command });
  },
});

const result = await generateText({
  model: __MODEL__,
  tools: { shell },
  toolsContext: {
    shell: { projectName: 'web-app' },
  },
  experimental_sandbox,
  prompt: 'List the project files.',
});
```

## 严格模式

启用后，支持严格 tool 调用的语言模型 provider 只会生成符合所定义 `inputSchema` 的有效 tool 调用。
这可提高 tool 调用的可靠性。
但并非所有 schema 在严格模式下都受支持，具体取决于 provider。

默认关闭严格模式。可对单个 tool 设置 `strict: true` 启用：

```ts
tool({
  description: 'Get the weather in a location',
  inputSchema: z.object({
    location: z.string(),
  }),
  strict: true, // Enable strict validation for this tool
  execute: async ({ location }) => ({
    // ...
  }),
});
```

<Note>
  并非所有 provider 或模型支持严格模式。不支持的会忽略该选项。
</Note>

## 输入示例

可为 tool 指定示例输入，引导模型如何结构化输入数据。
在 provider 支持时，当 JSON schema 无法完全表达预期用法或存在可选值时，输入示例很有帮助。

```ts
tool({
  description: 'Get the weather in a location',
  inputSchema: z.object({
    location: z.string().describe('The location to get the weather for'),
  }),
  inputExamples: [
    { input: { location: 'San Francisco' } },
    { input: { location: 'London' } },
  ],
  execute: async ({ location }) => {
    // ...
  },
});
```

<Note>
  仅 Anthropic provider 原生支持 tool 输入示例，其他 provider 会忽略该设置。
</Note>

## Tool 执行审批

默认情况下，带 `execute` 的 tool 会在模型调用时自动运行。在 `generateText`、`streamText` 或 `ToolLoopAgent` 上通过 `toolApproval` 配置审批。

`toolApproval` 可以是：

- 处理所有 tool 调用的 `GenericToolApprovalFunction`
- 按 tool 的状态映射和/或 `SingleToolApprovalFunction` 回调（函数返回 `undefined` 与 `'not-applicable'` 效果相同）

`tool()` 定义上的旧属性 `needsApproval` 已弃用。现有代码仍可用，新代码应将审批逻辑迁移到 `toolApproval`。

### 配置 `toolApproval`

```ts highlight="8-10"
const result = await generateText({
  model: __MODEL__,
  tools: { runCommand },
  toolApproval: {
    runCommand: 'user-approval',
  },
  prompt: 'Remove the most recent file in the downloads folder',
});
```

在按 tool 的对象形式中，每个键可为四种状态之一，可为字符串或带 `type` 字段的对象：

- `'not-applicable'`：不经过审批流程直接执行 tool。此为默认行为。
- `'approved'`：在输出中发出审批请求/响应部分以记录自动批准，然后执行 tool
- `'denied'`：在输出中发出审批请求/响应部分以记录自动拒绝，然后呈现被拒绝的 tool 输出
- `'user-approval'`：发出审批请求并等待明确响应

自动批准与拒绝也可返回对象以包含原因：

```ts
toolApproval: {
  runCommand: {
    type: 'denied',
    reason: 'blocked by policy',
  },
}
```

#### 通用 `toolApproval` 函数

将 `GenericToolApprovalFunction` 作为 `toolApproval` 传入，在一处处理所有 tool 调用。回调接收 `toolCall`、`tools`、`toolsContext`、`messages` 与 `runtimeContext`（与传给 `generateText` 或 `streamText` 的相同，类型为第二个 `ToolApprovalConfiguration` 类型参数；默认为 `Context`）。可返回 `undefined`，效果与 `'not-applicable'` 相同。

```ts highlight="4-10"
const result = await generateText({
  model: __MODEL__,
  tools: { runCommand },
  toolApproval: ({
    toolCall,
    tools,
    toolsContext,
    messages,
    runtimeContext,
  }) => {
    if (toolCall.toolName === 'runCommand' && !toolCall.dynamic) {
      // Inspect toolCall.input, cross-tool state, messages, or runtimeContext for policy.
      return 'user-approval';
    }
    return undefined; // or 'not-applicable'
  },
  prompt: 'Remove the most recent file in the downloads folder',
});
```

相同模式适用于 `streamText`，以及在需要跨所有 tool 统一策略时对 [`ToolLoopAgent`](/docs/agents/building-agents#tools) 的 `toolApproval`。

若使用按 tool 的对象，每个键可为 `SingleToolApprovalFunction`，接收 tool 输入及选项 `toolCallId`、`messages`、`toolContext`（与 tool 执行的 `context` 形状相同，不含 `abortSignal`）和 `runtimeContext`。返回 `undefined` 效果与 `'not-applicable'` 相同。

适用于执行命令、处理支付、修改数据等敏感或潜在危险操作的 tool。

### 工作原理

当 tool 需要人工审批时，`generateText` 与 `streamText` 不会暂停执行，而是完成并在结果内容中返回 `tool-approval-request` 部分。因此人工审批流程需要两次模型调用：第一次返回审批请求，第二次（收到审批响应后）执行 tool 或告知模型审批被拒绝。

人工审批来自 `toolApproval` 返回 `'user-approval'`。若 tool 应正常执行，使用 `'not-applicable'`、从审批函数返回 `undefined`，或完全省略设置。若 `toolApproval` 返回 `'approved'` 或 `'denied'` 或其对象形式，SDK 会在同一次生成中通过发出审批请求/响应部分自动记录该决定。自动批准或拒绝可提供 `reason`，该原因会包含在发出的审批响应中并可在 UI 渲染。

人工审批流程如下：

1. 使用 `toolApproval` 调用 `generateText` 或 `streamText`
2. 模型生成 tool 调用
3. 调用在 `result.content` 中返回 `tool-approval-request` 部分
4. 应用请求审批并收集用户决定
5. 将 `tool-approval-response` 加入 messages 数组
6. 用更新后的 messages 再次调用 `generateText` 或 `streamText`
7. 若批准，tool 运行并返回结果；若拒绝，模型看到拒绝并相应响应

### 处理审批请求

调用 `generateText` 或 `streamText` 后，在 `result.content` 中查找 `tool-approval-request` 部分：

```ts
import { type ModelMessage, generateText } from 'ai';

const messages: ModelMessage[] = [
  { role: 'user', content: 'Remove the most recent file' },
];
const result = await generateText({
  model: __MODEL__,
  tools: { runCommand },
  messages,
});

messages.push(...result.responseMessages);

for (const part of result.content) {
  if (part.type === 'tool-approval-request' && !part.isAutomatic) {
    console.log(part.approvalId); // Unique ID for this approval request
    console.log(part.toolCall); // Contains toolName, input, etc.
  }
}
```

响应时创建 `tool-approval-response` 并加入 messages：

```ts
import { type ToolApprovalResponse } from 'ai';

const approvals: ToolApprovalResponse[] = [];

for (const part of result.content) {
  if (part.type === 'tool-approval-request' && !part.isAutomatic) {
    const response: ToolApprovalResponse = {
      type: 'tool-approval-response',
      approvalId: part.approvalId,
      approved: true, // or false to deny
      reason: 'User confirmed the command', // Optional context for the model
    };
    approvals.push(response);
  }
}

// add approvals to messages
messages.push({ role: 'tool', content: approvals });
```

然后用更新后的 messages 再次调用 `generateText` 或 `streamText`。批准则执行 tool；拒绝则模型收到拒绝并可相应响应。

若 tool 应在输出中不含任何审批元数据即执行，使用 `'not-applicable'`、从审批函数返回 `undefined`，或省略 `toolApproval`。仅当希望结果包含自动 `tool-approval-request`（`isAutomatic: true`）及后续 `tool-approval-response` 以便检查或渲染决定、无需再次提示用户时，才使用 `'approved'`、`'denied'` 或其对象形式。

<Note>
  当 tool 执行被拒绝时，可考虑添加系统指令如「当 tool 执行未获批准时不要重试」，防止模型再次尝试相同调用。
</Note>

<Note>
  Provider 执行的 tool 在 provider 侧执行，不考虑 tool 审批设置。`toolApproval`（及已弃用的 `needsApproval`）仅控制 AI SDK 在本地执行的 tool。
</Note>

### 动态审批

可在按 tool 的 `toolApproval` 对象中提供异步函数，根据 tool 输入做审批决定：

```ts
const paymentTool = tool({
  description: 'Process a payment',
  inputSchema: z.object({
    amount: z.number(),
    recipient: z.string(),
  }),
  execute: async ({ amount, recipient }) => {
    return await processPayment(amount, recipient);
  },
});

const result = await generateText({
  model: __MODEL__,
  tools: {
    processPayment: paymentTool,
  },
  toolApproval: {
    processPayment: async ({ amount }) =>
      amount > 1000 ? 'user-approval' : undefined,
  },
  prompt: 'Send $1500 to the contractor',
});
```

本例仅超过 $1000 的交易需要审批，较小金额自动执行。

若希望在调用时返回 `'not-applicable'`、`undefined`（等同 `'not-applicable'`）、`'approved'`、`'denied'` 或 `'user-approval'`，而非在 tool 上定义默认值，可在按 tool 的 `toolApproval` 对象中使用 `SingleToolApprovalFunction`。自动批准与拒绝时，回调也可返回 `{ type, reason }`，例如 `{ type: 'denied', reason: 'blocked by policy' }`。若决定需要完整 `toolCall` 或同时涉及多个 tool，请传入 `GenericToolApprovalFunction` 作为 `toolApproval`，而非按 tool 映射。通用函数同样可返回 `undefined`。

### 配合 useChat 的 Tool 执行审批

使用 `useChat` 时，审批流程通过 UI 状态处理。详见 [Chatbot Tool 用法](/docs/ai-sdk-ui/chatbot-tool-usage#tool-execution-approval) 中如何用 `addToolApprovalResponse` 在 UI 处理审批。

## 多步调用（使用 stopWhen）

通过 `stopWhen` 可在 `generateText` 与 `streamText` 中启用多步调用。设置 `stopWhen` 且模型生成 tool 调用时，AI SDK 会传入 tool 结果触发新一次生成，直到不再有 tool 调用或满足停止条件。

AI SDK 提供多种内置停止条件：

- `isStepCount(count)` — 在指定步数后停止（默认：`isStepCount(20)`）
- `hasToolCall(...toolNames)` — 调用任一指定 tool 时停止
- `isLoopFinished()` — 永不触发，让循环自然结束

也可在数组中组合多个条件或创建自定义条件。详见[循环控制](/docs/agents/loop-control)。

<Note>
  仅当最后一步包含 tool 结果时才评估 `stopWhen` 条件。
</Note>

默认使用 `generateText` 或 `streamText` 时只触发单次生成，适用于可依赖模型训练数据回答的许多场景。但提供 tools 后，模型可选择生成普通文本或 tool 调用；若生成 tool 调用，该次生成即结束，该 step 完成。

你可能希望 tool 执行后模型再生成文本，例如在用户查询上下文中总结 tool 结果。许多场景还希望模型在一次响应中使用多个 tool，这就是多步调用的用途。

多步调用可类比与人对话：若对方常识（模型训练数据）中没有所需知识，可能需要查资料（使用 tool）才能回答。同样，模型可能需要调用 tool 获取回答问题所需信息，每次生成（tool 调用或文本生成）即一步。

### 示例

以下示例包含两步：

1. **Step 1**
   1. 将提示 `'What is the weather in San Francisco?'` 发给模型。
   1. 模型生成 tool 调用。
   1. 执行 tool 调用。
1. **Step 2**
   1. 将 tool 结果发给模型。
   1. 模型结合 tool 结果生成响应。

```ts highlight="18-19"
import { z } from 'zod';
import { generateText, tool, isStepCount } from 'ai';
__PROVIDER_IMPORT__;

const { text, steps } = await generateText({
  model: __MODEL__,
  tools: {
    weather: tool({
      description: 'Get the weather in a location',
      inputSchema: z.object({
        location: z.string().describe('The location to get the weather for'),
      }),
      execute: async ({ location }) => ({
        location,
        temperature: 72 + Math.floor(Math.random() * 21) - 10,
      }),
    }),
  },
  stopWhen: isStepCount(5), // stop after a maximum of 5 steps if tools were called
  prompt: 'What is the weather in San Francisco?',
});
```

<Note>可用类似方式使用 `streamText`。</Note>

### Step

To access intermediate tool calls and results, you can use the `steps` property in the result object
or the `streamText` `onFinish` callback.
It contains all the text, tool calls, tool results, per-step `performance`, and more from each step.

#### 示例: Extract tool results from all steps

```ts highlight="3,9-10"
import { generateText } from 'ai';
__PROVIDER_IMPORT__;

const { steps } = await generateText({
  model: __MODEL__,
  stopWhen: isStepCount(10),
  // ...
});

// extract all tool calls from the steps:
const allToolCalls = steps.flatMap(step => step.toolCalls);
```

### `onStepFinish` 回调

使用 `generateText` 或 `streamText` 时，可提供 `onStepFinish` 回调，
在 step 完成时触发，
即该 step 的全部文本增量、tool 调用与 tool 结果可用时。
多步时，每个 step 都会触发该回调。

回调接收 `stepNumber`（从 0 开始）以标识刚完成的 step：

```tsx highlight="5-8"
import { generateText } from 'ai';

const result = await generateText({
  // ...
  onStepFinish({
    stepNumber,
    text,
    toolCalls,
    toolResults,
    finishReason,
    usage,
    performance,
  }) {
    console.log(`Step ${stepNumber} finished (${finishReason})`, {
      usage,
      performance,
    });
    // your own logic, e.g. for saving the chat history or recording usage
  },
});
```

### Tool 执行生命周期回调

可使用 `onToolExecutionStart` 与 `onToolExecutionEnd` 观察 tool 执行。
这些回调在每个 tool 的 `execute` 函数前后调用，便于了解执行时机、输入、输出与错误：

```tsx highlight="5-14"
import { generateText } from 'ai';

const result = await generateText({
  // ... model, tools, prompt
  onToolExecutionStart({ toolCall }) {
    console.log(`Calling tool: ${toolCall.toolName}`, {
      toolCallId: toolCall.toolCallId,
      input: toolCall.input,
    });
  },
  onToolExecutionEnd({ toolCall, toolExecutionMs, toolOutput }) {
    if (toolOutput.type === 'tool-error') {
      console.error(
        `Tool ${toolCall.toolName} failed after ${toolExecutionMs}ms:`,
        toolOutput.error,
      );
    } else {
      console.log(
        `Tool ${toolCall.toolName} completed in ${toolExecutionMs}ms`,
        {
          output: toolOutput.output,
        },
      );
    }
  },
});
```

这些回调内抛出的错误会被静默捕获，不会中断生成流程。

### `prepareStep` 回调

`prepareStep` 回调在 step 开始前调用。

调用时传入以下参数：

- `model`: The model that was passed into `generateText`.
- `stopWhen`: The stopping condition that was passed into `generateText`.
- `stepNumber`: The number of the step that is being executed.
- `steps`: The steps that have been executed so far.
- `instructions`: The instructions that will be sent to the model for the current step. If `prepareStep` returns an `instructions` override, those instructions carry forward as the default for later steps.
- `initialInstructions`: The instructions that were passed into `generateText` or `streamText`.
- `messages`: The messages that will be sent to the model for the current step. Treat this as the loop's current message state. If `prepareStep` returns a `messages` override, those messages carry forward as the base for later steps.
- `initialMessages`: The messages that were passed into `generateText` or `streamText`.
- `responseMessages`: The accumulated assistant/tool response messages so far, including any tool results generated before the first model step from approved tool calls in the input messages.
- `runtimeContext`: The runtime context passed via the `runtimeContext` setting.
- `toolsContext`: The per-tool context map passed via the `toolsContext` setting.
- `experimental_sandbox`: The experimental sandbox passed via the `experimental_sandbox` setting.

可用以为 step 提供不同设置，包括修改输入 messages。

```tsx highlight="5-7"
import { generateText } from 'ai';

const result = await generateText({
  // ...
  prepareStep: async ({ model, stepNumber, steps, messages }) => {
    if (stepNumber === 0) {
      return {
        // use a different model for this step:
        model: modelForThisParticularStep,
        // force a tool choice for this step:
        toolChoice: { type: 'tool', toolName: 'tool1' },
        // limit the tools that are available for this step:
        activeTools: ['tool1'],
      };
    }

    // when nothing is returned, the default settings are used
  },
});
```

若返回 `instructions`，会延续到后续 step，直至 `prepareStep` 再次返回 `instructions` 或 `system` 覆盖。需要恢复或与原始调用顶层 instructions 比较时使用 `initialInstructions`。

#### 较长 Agent 循环的消息修改

在较长的 agent 循环中，可用 `messages` 参数修改后续 step 将使用的消息状态，尤其适用于上下文压缩，由你决定何时压缩。

`messages` 参数包含当前 step 的 messages。默认是 `initialMessages` 后接累积的 `responseMessages`。若先前 `prepareStep` 返回了 `messages`，后续 step 使用这些持久化 messages 加上一步的响应 messages。

需要原始输入 messages 时用 `initialMessages`；需要迄今离散的助手/tool 响应 messages 时用 `responseMessages`。

`pruneMessages` 辅助函数提供移除选定 messages 或 message 部分的内置方式，可在 `prepareStep` 内用于简单压缩策略。

```tsx
import { generateText, pruneMessages, type ModelMessage } from 'ai';

const COMPACTION_THRESHOLD = 100_000;

const estimateTokens = (messages: ModelMessage[]) => {
  return JSON.stringify(messages).length / 4;
};

const result = await generateText({
  // ...
  prepareStep: ({ messages }) => {
    if (estimateTokens(messages) > COMPACTION_THRESHOLD) {
      return {
        messages: pruneMessages({
          messages,
          reasoning: 'all',
          toolCalls: 'before-last-3-messages',
          emptyMessages: 'remove',
        }),
      };
    }
  },
});
```

本例使用估算 token 阈值，你可使用任意触发条件。关键行为是返回 `messages` 会改变后续 step 的消息状态。

返回的消息变更会在 step 间持久化。若希望每步 messages 来自原始输入加离散响应 messages 而非持久化状态，每次从 `initialMessages` 与 `responseMessages` 重建：

```tsx
prepareStep: ({ initialMessages, responseMessages, stepNumber }) => {
  if (stepNumber > 0) {
    return {
      messages: [...initialMessages, ...responseMessages.slice(-10)],
    };
  }
},
```

#### Step 配置的 Provider 选项

可在 `prepareStep` 中使用 `providerOptions` 为每步传入 provider 特定配置，例如 Anthropic 代码执行容器持久化：

```tsx
import { forwardAnthropicContainerIdFromLastStep } from '@ai-sdk/anthropic';

// Propagate container ID from previous step for code execution continuity
prepareStep: forwardAnthropicContainerIdFromLastStep,
```

## 响应 Messages

将生成的助手与 tool messages 加入对话历史是常见任务，
尤其在使用多步 tool 调用时。

`generateText` 与 `streamText` 均有 `responseMessages` 属性，可将助手与 tool messages 加入对话历史。
`streamText` 的 `onFinish` 回调中也可使用。

`responseMessages` 包含本次调用累积的响应 messages，为可加入对话历史的 `ModelMessage` 对象数组：

```ts
import { generateText, ModelMessage } from 'ai';

const messages: ModelMessage[] = [
  // ...
];

const { responseMessages } = await generateText({
  // ...
  messages,
});

// add the response messages to your conversation history:
messages.push(...responseMessages); // streamText: ...(await result.responseMessages)
```

## 动态 Tool

AI SDK Core 支持编译时未知 tool schema 的场景，适用于：

- MCP (Model Context Protocol) tools without schemas
- User-defined functions at runtime
- Tools loaded from external sources

### 使用 dynamicTool

`dynamicTool` 辅助函数创建输入/输出类型未知的 tool：

```ts
import { dynamicTool } from 'ai';
import { z } from 'zod';

const customTool = dynamicTool({
  description: 'Execute a custom function',
  inputSchema: z.object({}),
  execute: async input => {
    // input is typed as 'unknown'
    // You need to validate/cast it at runtime
    const { action, parameters } = input as any;

    // Execute your dynamic logic
    return { result: `Executed ${action}` };
  },
});
```

### 类型安全处理

同时使用静态与动态 tool 时，用 `dynamic` 标志进行类型收窄：

```ts
const result = await generateText({
  model: __MODEL__,
  tools: {
    // Static tool with known types
    weather: weatherTool,
    // Dynamic tool
    custom: dynamicTool({
      /* ... */
    }),
  },
  onStepFinish: ({ toolCalls, toolResults }) => {
    // Type-safe iteration
    for (const toolCall of toolCalls) {
      if (toolCall.dynamic) {
        // Dynamic tool: input is 'unknown'
        console.log('Dynamic:', toolCall.toolName, toolCall.input);
        continue;
      }

      // Static tool: full type inference
      switch (toolCall.toolName) {
        case 'weather':
          console.log(toolCall.input.location); // typed as string
          break;
      }
    }
  },
});
```

## 初步 Tool 结果

可返回包含多个结果的 `AsyncIterable`。
此时可迭代对象的最后一个值为最终 tool 结果。

可与生成器函数结合，例如在 tool 执行期间流式传输状态信息：

```ts
tool({
  description: 'Get the current weather.',
  inputSchema: z.object({
    location: z.string(),
  }),
  async *execute({ location }) {
    yield {
      status: 'loading' as const,
      text: `Getting weather for ${location}`,
      weather: undefined,
    };

    await new Promise(resolve => setTimeout(resolve, 3000));

    const temperature = 72 + Math.floor(Math.random() * 21) - 10;

    yield {
      status: 'success' as const,
      text: `The weather in ${location} is ${temperature}°F`,
      temperature,
    };
  },
});
```

## Tool 选择

可使用 `toolChoice` 设置影响何时选择 tool。
支持以下设置：

- `auto` (default): the model can choose whether and which tools to call.
- `required`: the model must call a tool. It can choose which tool to call.
- `none`: the model must not call tools
- `{ type: 'tool', toolName: string (typed) }`: the model must call the specified tool

```ts highlight="18"
import { z } from 'zod';
import { generateText, tool } from 'ai';
__PROVIDER_IMPORT__;

const result = await generateText({
  model: __MODEL__,
  tools: {
    weather: tool({
      description: 'Get the weather in a location',
      inputSchema: z.object({
        location: z.string().describe('The location to get the weather for'),
      }),
      execute: async ({ location }) => ({
        location,
        temperature: 72 + Math.floor(Math.random() * 21) - 10,
      }),
    }),
  },
  toolChoice: 'required', // force the model to call a tool
  prompt: 'What is the weather in San Francisco?',
});
```

## Tool 执行选项

调用 tool 时，第二个参数会传入额外选项。

### Tool 调用 ID

工具调用的 ID 会转发到工具执行中，
例如在向流数据发送与 tool 调用相关信息时使用。

```ts highlight="14-20"
import {
  streamText,
  tool,
  createUIMessageStream,
  createUIMessageStreamResponse,
} from 'ai';

export async function POST(req: Request) {
  const { messages } = await req.json();

  const stream = createUIMessageStream({
    execute: ({ writer }) => {
      const result = streamText({
        // ...
        messages,
        tools: {
          myTool: tool({
            // ...
            execute: async (args, { toolCallId }) => {
              // return e.g. custom status for tool call
              writer.write({
                type: 'data-tool-status',
                id: toolCallId,
                data: {
                  name: 'myTool',
                  status: 'in-progress',
                },
              });
              // ...
            },
          }),
        },
      });

      writer.merge(result.toUIMessageStream());
    },
  });

  return createUIMessageStreamResponse({ stream });
}
```

### Messages

The messages that were sent to the language model to initiate the response that contained the tool call are forwarded to the tool execution.
可在 `execute` 函数第二个参数中访问。
多步调用中，messages 包含先前所有 step 的文本、tool 调用与 tool 结果。

```ts highlight="8-9"
import { generateText, tool } from 'ai';

const result = await generateText({
  // ...
  tools: {
    myTool: tool({
      // ...
      execute: async (args, { messages }) => {
        // use the message history in e.g. calls to other language models
        return { ... };
      },
    }),
  },
});
```

### Abort 信号

The abort signals from `generateText` and `streamText` are forwarded to the tool execution.
可在 `execute` 第二个参数中访问，例如中止长时间计算或转发给 tool 内的 fetch 调用。

```ts highlight="6,11,14"
import { z } from 'zod';
import { generateText, tool } from 'ai';
__PROVIDER_IMPORT__;

const result = await generateText({
  model: __MODEL__,
  abortSignal: myAbortSignal, // signal that will be forwarded to tools
  tools: {
    weather: tool({
      description: 'Get the weather in a location',
      inputSchema: z.object({ location: z.string() }),
      execute: async ({ location }, { abortSignal }) => {
        return fetch(
          `https://api.weatherapi.com/v1/current.json?q=${location}`,
          { signal: abortSignal }, // forward the abort signal to fetch
        );
      },
    }),
  },
  prompt: 'What is the weather in San Francisco?',
});
```

### Experimental Sandbox

当 tool 需在执行环境中运行命令或代码时，向 `generateText`、`streamText` 或 `ToolLoopAgent` 调用传入 `experimental_sandbox`。experimental sandbox 可用于 tool 描述函数及 tool `execute` 的第二个参数。

<Note type="warning">
  This API is experimental and can change in patch releases. Passing an
  experimental sandbox does not sandbox the tool itself.
</Note>

Tool 代码仍在应用运行处执行。仅 tool 显式委托给 experimental sandbox 的操作（如 `experimental_sandbox.runCommand(...)`）在 experimental sandbox 环境中运行。

```ts highlight="7-15,21"
import { generateText, tool } from 'ai';
import { z } from 'zod';

const result = await generateText({
  model: __MODEL__,
  tools: {
    shell: tool({
      inputSchema: z.object({
        command: z.string(),
        workingDirectory: z.string().optional(),
      }),
      execute: async (
        { command, workingDirectory },
        { abortSignal, experimental_sandbox },
      ) => {
        if (!experimental_sandbox) {
          throw new Error('Experimental sandbox is not available');
        }

        return experimental_sandbox.runCommand({
          command,
          workingDirectory,
          abortSignal,
        });
      },
    }),
  },
  experimental_sandbox,
  prompt: 'List the files in the project.',
});
```

The experimental sandbox description is not added to the model prompt automatically. If the model should know about details such as the root directory, exposed ports, or public hostname, include `experimental_sandbox.description` in your system prompt,
instructions, user-visible context, or a tool description function.

`runCommand` also accepts an optional `workingDirectory` and `abortSignal`.
Use `workingDirectory` when a tool needs to run a command from a directory other
than the experimental sandbox implementation's default working directory. Forward
`abortSignal` from the tool execution options so the experimental sandbox can cancel the command if the overall operation is aborted or times out.

AI SDK 会转发你的 experimental sandbox 对象，但不会为你创建或隔离沙箱。

`Experimental_Sandbox` 接口仅是契约，隔离保证取决于实现或 experimental sandbox provider。使用 `child_process.exec` 与工作目录的本地实现并非安全边界，命令仍可访问该目录外路径。对不可信命令或用户导向的编码 agent，应使用真实隔离 provider、限制可用命令、设置命令超时，并对敏感操作结合 [tool 审批](#tool-execution-approval) 与 experimental sandbox。

### 运行时上下文

可通过 `runtimeContext` 从 `generateText` 或 `streamText` 传入任意运行时上下文。
该上下文在 `prepareStep` 中可用。

为避免与提示上下文或检索上下文混淆，文档将此功能称为 runtime context（运行时上下文）。

适用于租户信息、功能开关、会话数据等应影响 step 准备但不应嵌入提示的服务端状态。

Tool 执行上下文现已分离。若 tool 需要 API 密钥等服务端值，通过按 tool 名称键控的 `toolsContext` 传入。每个 tool 仅根据其 `contextSchema` 接收自己的类型化 `context`。

完整心智模型、示例、生命周期细节及在提示上下文、运行时上下文与 tool 上下文间选择的指南，见[运行时与 Tool 上下文](/docs/ai-sdk-core/runtime-and-tool-context)。

### Tool 上下文遥测

Tool 上下文常含 API 密钥、访问令牌或内部标识等服务端值。
使用 `telemetry.includeToolsContext` 将选定的顶层 context 属性包含进遥测集成：

```ts highlight="24-30"
const weatherTool = tool({
  description: 'Get the weather in a location',
  inputSchema: z.object({
    location: z.string(),
  }),
  contextSchema: z.object({
    weatherApiKey: z.string(),
    defaultUnit: z.enum(['celsius', 'fahrenheit']),
  }),
  execute: async ({ location }, { context }) => {
    return fetchWeather({
      location,
      apiKey: context.weatherApiKey,
      unit: context.defaultUnit,
    });
  },
});

const result = await generateText({
  model: __MODEL__,
  tools: { weather: weatherTool },
  toolsContext: {
    weather: {
      weatherApiKey: process.env.WEATHER_API_KEY,
      defaultUnit: 'fahrenheit',
    },
  },
  prompt: 'What is the weather in San Francisco?',
  telemetry: {
    includeToolsContext: {
      weather: {
        defaultUnit: true,
      },
    },
  },
});
```

Telemetry integrations receive the `weather` tool context as `{ defaultUnit: 'fahrenheit' }`.
Properties set to `false` or omitted are excluded. If `telemetry.includeToolsContext` is omitted, no tool context properties are included.

<Note>
  `telemetry.includeToolsContext` only filters telemetry integrations. Tool
  execution, lifecycle callbacks, and returned results still receive the full
  typed tool context. See [Runtime and Tool
  Context](/docs/ai-sdk-core/runtime-and-tool-context) for how tool context
  flows through execution and telemetry.
</Note>

## Tool 输入生命周期钩子

以下 tool 输入生命周期钩子可用：

- **`onInputStart`**: Called when the model starts generating the input (arguments) for the tool call
- **`onInputDelta`**: Called for each chunk of text as the input is streamed
- **`onInputAvailable`**: Called when the complete input is available and validated

`onInputStart` 与 `onInputDelta` 仅在流式上下文（使用 `streamText`）中调用，使用 `generateText` 时不调用。

### 示例

```ts highlight="15-23"
import { streamText, tool } from 'ai';
__PROVIDER_IMPORT__;
import { z } from 'zod';

const result = streamText({
  model: __MODEL__,
  tools: {
    getWeather: tool({
      description: 'Get the weather in a location',
      inputSchema: z.object({
        location: z.string().describe('The location to get the weather for'),
      }),
      execute: async ({ location }) => ({
        temperature: 72 + Math.floor(Math.random() * 21) - 10,
      }),
      onInputStart: () => {
        console.log('Tool call starting');
      },
      onInputDelta: ({ inputTextDelta }) => {
        console.log('Received input chunk:', inputTextDelta);
      },
      onInputAvailable: ({ input }) => {
        console.log('Complete input:', input);
      },
    }),
  },
  prompt: 'What is the weather in San Francisco?',
});
```

## 类型

模块化代码常需定义类型以保证类型安全与复用。AI SDK 为此提供多种 tool、tool 调用与 tool 结果的辅助类型。

可在与 `streamText` 或 `generateText` 无直接关系的代码部分，为变量、函数参数与返回类型提供强类型。

每个 tool 调用按所调用的 tool 类型化为 `ToolCall<NAME extends string, ARGS>`。
同样，tool 结果类型化为 `ToolResult<NAME extends string, ARGS, RESULT>`。

`streamText` 与 `generateText` 中的 tools 定义为 `ToolSet`。
类型推断辅助 `TypedToolCall<TOOLS extends ToolSet>` 与 `TypedToolResult<TOOLS extends ToolSet>` 可从 tools 提取 tool 调用与结果类型。

```ts highlight="18-19,23-24"
import { TypedToolCall, TypedToolResult, generateText, tool } from 'ai';
__PROVIDER_IMPORT__;
import { z } from 'zod';

const myToolSet = {
  firstTool: tool({
    description: 'Greets the user',
    inputSchema: z.object({ name: z.string() }),
    execute: async ({ name }) => `Hello, ${name}!`,
  }),
  secondTool: tool({
    description: 'Tells the user their age',
    inputSchema: z.object({ age: z.number() }),
    execute: async ({ age }) => `You are ${age} years old!`,
  }),
};

type MyToolCall = TypedToolCall<typeof myToolSet>;
type MyToolResult = TypedToolResult<typeof myToolSet>;

async function generateSomething(prompt: string): Promise<{
  text: string;
  toolCalls: Array<MyToolCall>; // typed tool calls
  toolResults: Array<MyToolResult>; // typed tool results
}> {
  return generateText({
    model: __MODEL__,
    tools: myToolSet,
    prompt,
  });
}
```

## 错误处理

AI SDK 有三种与 tool 调用相关的错误：

- [`NoSuchToolError`](/docs/reference/ai-sdk-errors/ai-no-such-tool-error): the model tries to call a tool that is not defined in the tools object
- [`InvalidToolInputError`](/docs/reference/ai-sdk-errors/ai-invalid-tool-input-error): the model calls a tool with inputs that do not match the tool's input schema
- [`ToolCallRepairError`](/docs/reference/ai-sdk-errors/ai-tool-call-repair-error): an error that occurred during tool call repair

当 tool 执行失败（tool 的 `execute` 抛出错误）时，AI SDK 将其作为 `tool-error` 内容部分加入，以在多步场景中支持自动 LLM 往返。

### `generateText`

`generateText` 会对 tool schema 校验等问题及其他错误抛出异常，可用 `try`/`catch` 处理。Tool 执行错误在结果 step 中表现为 `tool-error` 部分：

```ts
try {
  const result = await generateText({
    //...
  });
} catch (error) {
  if (NoSuchToolError.isInstance(error)) {
    // handle the no such tool error
  } else if (InvalidToolInputError.isInstance(error)) {
    // handle the invalid tool inputs error
  } else {
    // handle other errors
  }
}
```

Tool 执行错误可在结果 step 中获取：

```ts
const { steps } = await generateText({
  // ...
});

// check for tool errors in the steps
const toolErrors = steps.flatMap(step =>
  step.content.filter(part => part.type === 'tool-error'),
);

toolErrors.forEach(toolError => {
  console.log('Tool error:', toolError.error);
  console.log('Tool name:', toolError.toolName);
  console.log('Tool input:', toolError.input);
});
```

### `streamText`

`streamText` 将错误作为完整流的一部分发送。Tool 执行错误表现为 `tool-error` 部分，其他错误为 `error` 部分。

使用 `toUIMessageStreamResponse` 时，可传入 `onError` 从错误部分提取错误消息并作为流响应的一部分转发：

```ts
const result = streamText({
  // ...
});

return result.toUIMessageStreamResponse({
  onError: error => {
    if (NoSuchToolError.isInstance(error)) {
      return 'The model tried to call a unknown tool.';
    } else if (InvalidToolInputError.isInstance(error)) {
      return 'The model called a tool with invalid inputs.';
    } else {
      return 'An unknown error occurred.';
    }
  },
});
```

## Tool 调用修复

<Note type="warning">
  tool 调用修复功能为实验性，未来可能变更。
</Note>

语言模型有时无法生成有效 tool 调用，尤其在 input schema 复杂或模型较小时。

若使用多步，失败的 tool 调用会在下一步发回 LLM 以有机会修正。
但你可能希望控制无效 tool 调用的修复方式，而无需额外污染消息历史的 step。

可使用 `experimental_repairToolCall` 通过自定义函数尝试修复 tool 调用。

可用不同策略修复 tool 调用：

- Use a model with structured outputs to generate the inputs.
- Send the messages, instructions, and tool schema to a stronger model to generate the inputs.
- Provide more specific repair instructions based on which tool was called.

### 示例: Use a model with structured outputs for repair

```ts
import { openai } from '@ai-sdk/openai';
import { generateText, NoSuchToolError, Output, tool } from 'ai';

const result = await generateText({
  model,
  tools,
  prompt,

  experimental_repairToolCall: async ({
    toolCall,
    tools,
    inputSchema,
    error,
  }) => {
    if (NoSuchToolError.isInstance(error)) {
      return null; // do not attempt to fix invalid tool names
    }

    const tool = tools[toolCall.toolName as keyof typeof tools];

    const { output: repairedArgs } = await generateText({
      model: __MODEL__,
      output: Output.object({ schema: tool.inputSchema }),
      prompt: [
        `The model tried to call the tool "${toolCall.toolName}"` +
          ` with the following inputs:`,
        JSON.stringify(toolCall.input),
        `The tool accepts the following schema:`,
        JSON.stringify(await inputSchema({ toolName: toolCall.toolName })),
        'Please fix the inputs.',
      ].join('\n'),
    });

    return { ...toolCall, input: JSON.stringify(repairedArgs) };
  },
});
```

### 示例: Use the re-ask strategy for repair

```ts
import { openai } from '@ai-sdk/openai';
import { generateText, NoSuchToolError, tool } from 'ai';

const result = await generateText({
  model,
  tools,
  prompt,

  experimental_repairToolCall: async ({
    toolCall,
    tools,
    error,
    messages,
    instructions,
  }) => {
    const result = await generateText({
      model,
      instructions,
      messages: [
        ...messages,
        {
          role: 'assistant',
          content: [
            {
              type: 'tool-call',
              toolCallId: toolCall.toolCallId,
              toolName: toolCall.toolName,
              input: toolCall.input,
            },
          ],
        },
        {
          role: 'tool' as const,
          content: [
            {
              type: 'tool-result',
              toolCallId: toolCall.toolCallId,
              toolName: toolCall.toolName,
              output: error.message,
            },
          ],
        },
      ],
      tools,
    });

    const newToolCall = result.toolCalls.find(
      newToolCall => newToolCall.toolName === toolCall.toolName,
    );

    return newToolCall != null
      ? {
          type: 'tool-call' as const,
          toolCallId: toolCall.toolCallId,
          toolName: toolCall.toolName,
          input: JSON.stringify(newToolCall.input),
        }
      : null;
  },
});
```

## 活动 Tool

语言模型一次只能处理有限数量的 tool，取决于模型。
为在大量使用 tool 时仍保持静态类型并同时限制模型可用 tool，AI SDK 提供 `activeTools` 属性。

其为当前活动的 tool 名称数组。
默认值为 `undefined`，表示所有 tool 均活动。

```ts highlight="7"
import { openai } from '@ai-sdk/openai';
import { generateText } from 'ai';
__PROVIDER_IMPORT__;

const { text } = await generateText({
  model: __MODEL__,
  tools: myToolSet,
  activeTools: ['firstTool'],
});
```

## 多模态 Tool 结果

<Note type="warning">
  Multi-modal tool results are experimental and supported by Anthropic, OpenAI,
  and Google (Gemini 3 models).
</Note>

Google 请使用 base64 内联数据 file 部分（`{ type: 'file', mediaType, data: { type: 'data', data } }`）或 URL 风格 file 部分中的 base64 `data:` URL（`{ type: 'file', mediaType, data: { type: 'url', url: new URL('data:...') } }`）。tool 结果 URL 部分不支持远程 HTTP(S) URL。

要将多模态 tool 结果（如截图）发回模型，需转换为特定格式。

AI SDK Core tools have an optional `toModelOutput` function
that converts the tool result into a content part.

以下示例将截图转换为内容部分：

```ts highlight="22-27"
const result = await generateText({
  model: __MODEL__,
  tools: {
    computer: anthropic.tools.computer_20241022({
      // ...
      async execute({ action, coordinate, text }) {
        switch (action) {
          case 'screenshot': {
            return {
              type: 'file',
              mediaType: 'image',
              data: fs
                .readFileSync('./data/screenshot-editor.png')
                .toString('base64'),
            };
          }
          default: {
            return `executed ${action}`;
          }
        }
      },

      // map to tool result content for LLM consumption:
      toModelOutput({ output }) {
        return {
          type: 'content',
          value:
            typeof output === 'string'
              ? [{ type: 'text', text: output }]
              : [
                  {
                    type: 'file',
                    mediaType: 'image/png',
                    data: { type: 'data', data: output.data },
                  },
                ],
        };
      },
    }),
  },
  // ...
});
```

## MCP Tool

AI SDK 支持连接 Model Context Protocol（MCP）服务器以使用其 tool。
MCP 使 AI 应用可通过标准化接口发现并跨服务使用 tool。

### AI SDK Tool 与 MCP Tool

生产应用通常应定义自己的 AI SDK tool，以获得完全控制、类型安全与最佳性能。MCP tool 最适合快速开发迭代与用户自带 tool 的场景。

| Aspect                 | AI SDK Tools                                              | MCP Tools                                             |
| ---------------------- | --------------------------------------------------------- | ----------------------------------------------------- |
| **Type Safety**        | Full static typing end-to-end                             | Dynamic discovery at runtime                          |
| **Execution**          | Same process as your request (low latency)                | Separate server (network overhead)                    |
| **Prompt Control**     | Full control over descriptions and schemas                | Controlled by MCP server owner                        |
| **Schema Control**     | You define and optimize for your model                    | Controlled by MCP server owner                        |
| **Version Management** | Full visibility over updates                              | Can update independently (version skew risk)          |
| **Authentication**     | Same process, no additional auth required                 | Separate server introduces additional auth complexity |
| **Best For**           | Production applications requiring control and performance | Development iteration, user-provided tools            |
