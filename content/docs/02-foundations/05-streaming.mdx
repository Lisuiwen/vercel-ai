# 流式传输

流式对话文本 UI（如 ChatGPT）在过去几个月迅速流行。本节探讨流式与阻塞界面的利弊。

[大语言模型（LLM）](/docs/foundations/overview#large-language-models)非常强大。但在生成长输出时，相比你可能习惯的延迟，它们可能非常慢。若构建传统阻塞 UI，用户可能要盯着加载动画 5、10 甚至 40 秒，等待整个 LLM 响应生成完毕。这在聊天机器人等对话应用中会导致较差体验。流式 UI 可通过**在部分内容可用时即显示**来缓解该问题。

## 真实示例

以下两个示例说明流式 UI 如何在真实场景中改善体验——第一个使用阻塞 UI，第二个使用流式 UI。

### 流式 UI

<InlinePrompt initialInput="Come up with the first 200 characters of the first book in the Harry Potter series." />

可见，流式 UI 能比阻塞 UI 更快开始显示响应。因为阻塞 UI 必须等待整个响应生成完毕才能显示任何内容，而流式 UI 可在部分内容可用时即显示。

虽然流式界面能显著改善体验（尤其在使用较大语言模型时），但并非总是必要或有益。若使用更小、更快的模型即可实现目标功能且无需流式，该路径往往能带来更简单、更易管理的开发流程。

不过，无论模型速度如何，AI SDK 都旨在尽可能简化流式 UI 的实现。在下方示例中，我们使用 SDK 的 [`streamText`](/docs/reference/ai-sdk-core/stream-text) 函数，用不到 10 行代码实现文本流式生成：

```ts
import { streamText } from 'ai';
__PROVIDER_IMPORT__;

const { textStream } = streamText({
  model: __MODEL__,
  prompt: 'Write a poem about embedding models.',
});

for await (const textPart of textStream) {
  console.log(textPart);
}
```
