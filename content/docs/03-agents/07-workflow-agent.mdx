# WorkflowAgent

`@ai-sdk/workflow` 中的 `WorkflowAgent` 专为构建在 [workflow](https://vercel.com/docs/workflow) 内运行的**持久、可恢复 Agent** 而设计。它提供与 [`ToolLoopAgent`](/docs/agents/building-agents) 相同的 Agent 循环，但增加了自动状态持久化、工具 schema 序列化以及在工作流步骤边界间存活的内置工具审批流程。

## 为何需要持久 Agent？

标准 `ToolLoopAgent` 完全在内存中运行——如果进程崩溃，所有进度都会丢失。对于进行多次工具调用的生产 Agent，这会带来问题：

- **有状态** — 长时间运行的 Agent 循环需要在进程边界间持久化状态
- **可恢复** — 如果某步失败，你希望从上次检查点重试，而不是从头开始
- **人在回路中** — 需要用户审批的工具必须暂停 Agent 并在之后恢复
- **可观测性** — 每次工具调用作为独立的工作流步骤运行，在仪表板中可见

`WorkflowAgent` 通过在 workflow 内运行来解决这些问题，其中每次工具执行都是带有自动重试的持久步骤。

## 何时使用 WorkflowAgent vs ToolLoopAgent

|                         | ToolLoopAgent             | WorkflowAgent                                   |
| ----------------------- | ------------------------- | ----------------------------------------------- |
| **包**             | `ai`                      | `@ai-sdk/workflow`                              |
| **运行时**             | 内存中                 | Workflow                                        |
| **持久性**          | 崩溃后丢失             | 在重启后存活               |
| **工具重试**        | 手动                    | 自动（通过 workflow 步骤）                  |
| **人工审批**      | 内置                  | 内置 + 在挂起后存活                  |
| **`generate()` 方法** | 可用                 | 不可用                                   |
| **`stream()` 方法**   | 可用                 | 主要 API                                     |
| **流输出**       | `streamText` 返回值 | 带 `ModelCallStreamPart` 的 `writable` 参数 |

对于不需要持久性的更简单用例，请使用 `ai` 包中的 [`ToolLoopAgent`](/docs/agents/building-agents)。

## 安装

```bash
npm install @ai-sdk/workflow workflow
```

`@ai-sdk/workflow` 需要 `ai` 包和 `zod` 作为 peer dependencies。`workflow` 包提供 Workflow DevKit 运行时（`getWritable`、`'use workflow'`、`'use step'`）。

## 创建 WorkflowAgent

通过使用模型、instructions 和 tools 实例化 `WorkflowAgent` 类来定义 Agent：

```ts
import { WorkflowAgent } from '@ai-sdk/workflow';
import { tool } from 'ai';
import { z } from 'zod';

const agent = new WorkflowAgent({
  model: 'anthropic/claude-sonnet-4-6',
  instructions: 'You are a helpful assistant.',
  tools: {
    weather: tool({
      description: 'Get weather for a location',
      inputSchema: z.object({
        location: z.string(),
      }),
      execute: async ({ location }) => ({
        location,
        temperature: 72,
      }),
    }),
  },
});
```

### 模型解析

`model` 参数接受两种形式：

```ts
// String — AI Gateway model ID
new WorkflowAgent({ model: 'anthropic/claude-sonnet-4-6' });

// Provider instance
import { openai } from '@ai-sdk/openai';
new WorkflowAgent({ model: openai('gpt-4o') });
```

## 在工作流中使用 Agent

`WorkflowAgent` 设计为在工作流函数内运行。关键集成点是：

1. 用 `'use workflow'` 标记你的函数
2. 将 `getWritable()` 传递给 Agent 的 `stream()` 方法
3. 从 API 路由启动工作流

### 端到端示例

```ts filename="workflow/agent-chat.ts"
import { WorkflowAgent, type ModelCallStreamPart } from '@ai-sdk/workflow';
import { convertToModelMessages, tool, type UIMessage } from 'ai';
import { getWritable } from 'workflow';
import { z } from 'zod';

export async function chat(messages: UIMessage[]) {
  'use workflow';

  const modelMessages = await convertToModelMessages(messages);

  const agent = new WorkflowAgent({
    model: 'anthropic/claude-sonnet-4-6',
    instructions: 'You are a flight booking assistant.',
    tools: {
      searchFlights: tool({
        description: 'Search for available flights',
        inputSchema: z.object({
          origin: z.string(),
          destination: z.string(),
          date: z.string(),
        }),
        execute: searchFlightsStep,
      }),
      bookFlight: tool({
        description: 'Book a specific flight',
        inputSchema: z.object({
          flightId: z.string(),
          passengerName: z.string(),
        }),
        execute: bookFlightStep,
      }),
    },
  });

  const result = await agent.stream({
    messages: modelMessages,
    writable: getWritable<ModelCallStreamPart>(),
  });

  return { messages: result.messages };
}
```

```ts filename="app/api/chat/route.ts"
import { createModelCallToUIChunkTransform } from '@ai-sdk/workflow';
import { createUIMessageStreamResponse, type UIMessage } from 'ai';
import { start } from 'workflow/api';
import { chat } from '@/workflow/agent-chat';

export async function POST(request: Request) {
  const { messages }: { messages: UIMessage[] } = await request.json();

  const run = await start(chat, [messages]);

  return createUIMessageStreamResponse({
    stream: run.readable.pipeThrough(createModelCallToUIChunkTransform()),
  });
}
```

### 消息转换

`WorkflowAgent.stream()` 期望 `ModelMessage[]`，而非 `UIMessage[]`。从客户端（通过 `useChat`）接收消息时，请先转换：

```ts
import { convertToModelMessages, type UIMessage } from 'ai';

export async function chat(messages: UIMessage[]) {
  'use workflow';

  const modelMessages = await convertToModelMessages(messages);

  const result = await agent.stream({
    messages: modelMessages,
    // ...
  });
}
```

### Writable 流

与 `ToolLoopAgent` 中你消费返回的流不同，`WorkflowAgent` 将原始 `ModelCallStreamPart` 块写入 workflow 运行时通过 `getWritable()` 提供的 `writable` 流。在响应边界处，使用 `createModelCallToUIChunkTransform()` 将它们转换为客户端的 `UIMessageChunk` 对象：

```ts
import { createModelCallToUIChunkTransform } from '@ai-sdk/workflow';
import { createUIMessageStreamResponse } from 'ai';

// Convert raw model stream parts → UI message chunks
return createUIMessageStreamResponse({
  stream: run.readable.pipeThrough(createModelCallToUIChunkTransform()),
});
```

## 使用 WorkflowChatTransport 实现可恢复流式传输

工作流函数可能超时或因网络故障中断。`WorkflowChatTransport` 是 [`ChatTransport`](/docs/ai-sdk-ui/transport) 实现，可自动处理这些中断——它检测流在没有 `finish` 事件时结束，并重新连接以从中断处恢复。

```tsx filename="app/page.tsx"
'use client';

import { useChat } from '@ai-sdk/react';
import { WorkflowChatTransport } from '@ai-sdk/workflow';
import { useMemo } from 'react';

export default function Chat() {
  const transport = useMemo(
    () =>
      new WorkflowChatTransport({
        api: '/api/chat',
        maxConsecutiveErrors: 5,
        initialStartIndex: -50, // On page refresh, fetch last 50 chunks
      }),
    [],
  );

  const { messages, sendMessage } = useChat({ transport });

  // ... render chat UI
}
```

该 transport 要求你的 POST 端点返回 `x-workflow-run-id` 响应头，并在 `{api}/{runId}/stream` 提供 GET 端点用于重连：

```ts filename="app/api/chat/route.ts"
import { createModelCallToUIChunkTransform } from '@ai-sdk/workflow';
import { createUIMessageStreamResponse, type UIMessage } from 'ai';
import { start } from 'workflow/api';
import { chat } from '@/workflow/agent-chat';

export async function POST(request: Request) {
  const { messages }: { messages: UIMessage[] } = await request.json();
  const run = await start(chat, [messages]);

  return createUIMessageStreamResponse({
    stream: run.readable.pipeThrough(createModelCallToUIChunkTransform()),
    headers: {
      'x-workflow-run-id': run.runId,
    },
  });
}
```

```ts filename="app/api/chat/[runId]/stream/route.ts"
import { createModelCallToUIChunkTransform } from '@ai-sdk/workflow';
import type { NextRequest } from 'next/server';
import { getRun } from 'workflow/api';

export async function GET(
  request: NextRequest,
  { params }: { params: Promise<{ runId: string }> },
) {
  const { runId } = await params;
  const startIndex = Number(
    new URL(request.url).searchParams.get('startIndex') ?? '0',
  );

  const run = await getRun(runId);
  const readable = run
    .getReadable({ startIndex })
    .pipeThrough(createModelCallToUIChunkTransform());

  return new Response(readable, {
    headers: {
      'Content-Type': 'text/event-stream',
      'Cache-Control': 'no-cache',
      Connection: 'keep-alive',
      'x-workflow-run-id': runId,
    },
  });
}
```

完整 API 参考请参阅 [`WorkflowChatTransport`](/docs/reference/ai-sdk-workflow/workflow-chat-transport)。

## 工具作为工作流步骤

用 `'use step'` 标记工具 execute 函数，使其成为持久工作流步骤。这为每次工具调用提供：

- **自动重试** — 失败的工具调用自动重试（默认：3 次）
- **持久性** — 结果在进程重启后存活
- **可观测性** — 每次工具调用在工作流仪表板中显示为独立步骤

```ts
async function searchFlightsStep(input: {
  origin: string;
  destination: string;
  date: string;
}) {
  'use step';
  const response = await fetch(`https://api.flights.example/search?...`);
  return response.json();
}

async function bookFlightStep(input: {
  flightId: string;
  passengerName: string;
}) {
  'use step';
  const response = await fetch('https://api.flights.example/book', {
    method: 'POST',
    body: JSON.stringify(input),
  });
  return response.json();
}
```

没有 `'use step'` 的工具仍可工作，但作为常规内存函数运行，没有持久性保证。

## 工具审批

对于 `WorkflowAgent`，人工审批在工具定义上使用 `needsApproval` 配置。
这特定于 `WorkflowAgent`；对于 `generateText`、`streamText` 和 `ToolLoopAgent`，请改用 `toolApproval`。
当工作流工具设置了 `needsApproval` 时，Agent 暂停并向 writable 流发出审批请求。
工作流挂起直到用户批准或拒绝：

```ts
const agent = new WorkflowAgent({
  model: 'anthropic/claude-sonnet-4-6',
  tools: {
    bookFlight: tool({
      description: 'Book a flight',
      inputSchema: z.object({
        flightId: z.string(),
        passengerName: z.string(),
      }),
      needsApproval: true, // Always require approval
      execute: bookFlightStep,
    }),
    cancelBooking: tool({
      description: 'Cancel a booking',
      inputSchema: z.object({ bookingId: z.string() }),
      // Conditional approval based on input
      needsApproval: async input => {
        return input.bookingId.startsWith('VIP-');
      },
      execute: cancelBookingStep,
    }),
  },
});
```

由于工作流是持久的，审批请求在进程重启后仍然存活——用户可以在数小时后批准，Agent 将恢复。

## 循环控制

控制 Agent 可以执行的步数：

```ts
import { isStepCount } from 'ai';

const result = await agent.stream({
  messages,
  stopWhen: isStepCount(10), // Stop after 10 LLM calls
});
```

如果你希望 Agent 持续运行直到自然完成工具调用，也可以使用 `isLoopFinished()`：

```ts
import { isLoopFinished } from 'ai';

const result = await agent.stream({
  messages,
  stopWhen: isLoopFinished(),
});
```

`isLoopFinished()` 让 Agent 运行直到所有工具调用完成，但你仍应配合 `maxSteps` 使用以避免失控循环。请参阅 https://ai-sdk.dev/v7/docs/reference/ai-sdk-core/loop-finished#isloopfinished。

默认情况下，Agent 循环直到模型停止调用工具（无最大步数限制）。

## 结构化输出

使用 `Output` 将 Agent 响应解析为类型化对象：

```ts
import { Output } from '@ai-sdk/workflow';
import { z } from 'zod';

const result = await agent.stream({
  messages,
  output: Output.object({
    schema: z.object({
      sentiment: z.enum(['positive', 'neutral', 'negative']),
      summary: z.string(),
    }),
  }),
});

console.log(result.output); // { sentiment: 'positive', summary: '...' }
```

## 配置选项

`WorkflowAgent` 接受与 `ToolLoopAgent` 相同的生成设置（`temperature`、`maxOutputTokens`、`topP` 等），以及工作流特定选项。

### runtimeContext 和 toolsContext

在不放入提示词的情况下，通过 Agent 循环传递服务端状态。请使用这些选项替代之前的 `experimental_context` 选项。

- `runtimeContext` 是流经 `prepareStep`、生命周期回调和 `onFinish` 的共享 Agent 状态。将其视为不可变；从 `prepareStep` 返回新值以更新当前及后续步骤。
- `toolsContext` 是按工具名称键控的每工具映射。每个工具的 `execute` 仅将其自己的已验证条目视为 `context`。声明了 `contextSchema` 的工具会在执行前根据 schema 验证其条目。

```ts filename="workflow/agent-chat.ts"
import { WorkflowAgent } from '@ai-sdk/workflow';
import { tool } from 'ai';
import { z } from 'zod';

const agent = new WorkflowAgent({
  model: 'anthropic/claude-sonnet-4-6',
  tools: {
    weather: tool({
      description: 'Get the weather for a city.',
      inputSchema: z.object({ city: z.string() }),
      contextSchema: z.object({
        defaultUnit: z.enum(['celsius', 'fahrenheit']),
      }),
      execute: async ({ city }, { context }) => ({
        city,
        unit: context.defaultUnit,
      }),
    }),
  },

  // Shared agent state — available in `prepareStep`, lifecycle callbacks, and `onFinish`.
  runtimeContext: {
    tenantId: 'tenant_123',
    requestId: 'req_abc',
    plan: 'enterprise',
  },

  // Per-tool context — each tool sees only its own validated entry.
  toolsContext: {
    weather: { defaultUnit: 'celsius' },
  },

  prepareStep: ({ runtimeContext }) => {
    if (runtimeContext.plan === 'enterprise') {
      return { temperature: 0.2 };
    }
    return {};
  },
});
```

`runtimeContext` 和 `toolsContext` 也可以在每次调用 `stream()` 时传递，它们会覆盖构造函数级别的默认值。

因为 `WorkflowAgent` 在 Workflow 运行时内运行，上下文值可能在工作流和步骤边界间被持久化和重放。保持 `runtimeContext`、`toolsContext` 以及从 `prepareStep` 返回的任何上下文值可序列化。使用纯数据，如字符串、数字、布尔值、数组、普通对象、日期、URL、Map、Set 以及其他 Workflow 支持的结构化数据。不要在上下文中放置函数、类实例、Symbol、`WeakMap`、`WeakSet`、数据库客户端或 SDK 客户端。改为传递标识符或配置数据，并在步骤函数内重新创建不可序列化的资源。

这与 `ToolLoopAgent` 不同，后者在内存中运行，可在单个进程的生命周期内携带更丰富的 JavaScript 值。使用 `WorkflowAgent` 时，将上下文视为持久数据可使工作流重放和步骤执行可靠。

### prepareCall

在 Agent 循环开始前调用一次。使用它根据运行时上下文转换 model、instructions 或其他设置：

```ts
const agent = new WorkflowAgent({
  model: 'anthropic/claude-sonnet-4-6',
  prepareCall: async ({ model, tools, messages }) => {
    return {
      instructions: `Current time: ${new Date().toISOString()}`,
    };
  },
});
```

### prepareStep

在每一步（LLM 调用）之前调用。使用它修改设置、管理上下文或动态注入消息：

```ts
const agent = new WorkflowAgent({
  model: 'anthropic/claude-sonnet-4-6',
  prepareStep: async ({ stepNumber, messages }) => {
    if (stepNumber > 5) {
      return { toolChoice: 'none' }; // Force text response after 5 steps
    }
    return {};
  },
});
```

`prepareCall` 和 `prepareStep` 也可以在 `stream()` 中按调用传递。

## 生命周期回调

Agent 提供用于日志、可观测性和自定义遥测的生命周期回调。所有回调可以在构造函数（Agent 范围）或 `stream()`（每次调用）中定义。当两者都提供时，两者都会触发（先构造函数）：

```ts
const agent = new WorkflowAgent({
  model: 'anthropic/claude-sonnet-4-6',

  experimental_onStart({ modelId, messages }) {
    console.log('Agent started');
  },

  experimental_onStepStart({ stepNumber }) {
    console.log(`Step ${stepNumber} starting`);
  },

  onToolExecutionStart({ toolCall }) {
    console.log(`Calling tool: ${toolCall.toolName}`);
  },

  onToolExecutionEnd({ toolCall, toolOutput }) {
    console.log(`Tool finished: ${toolCall.toolName}`);
  },

  onStepFinish({ usage, finishReason }) {
    console.log('Step done:', { finishReason });
  },

  onFinish({ steps, totalUsage }) {
    console.log(`Completed in ${steps.length} steps`);
  },
});
```

## 从 `DurableAgent` 迁移

`WorkflowAgent` 替代 Workflow DevKit 的 [`DurableAgent`](https://workflow-sdk.dev/docs/api-reference/workflow-ai/durable-agent)。两者共享相同的核心思想——在 workflow 内运行的持久 Agent 循环——但 `WorkflowAgent` 将类移入 AI SDK，收紧类型，并引入一流的工具审批。如果你今天正在使用 `DurableAgent`，请按照以下步骤切换。

### 更改导入和类名

`DurableAgent` 从 `workflow/ai` 导出。`WorkflowAgent` 与 helpers 一起从 `@ai-sdk/workflow` 导出。

```diff
- import { DurableAgent } from 'workflow/ai';
+ import { WorkflowAgent, type ModelCallStreamPart } from '@ai-sdk/workflow';

- const agent = new DurableAgent({
+ const agent = new WorkflowAgent({
    model: 'anthropic/claude-sonnet-4-6',
    instructions: 'You are a helpful assistant.',
    tools: { /* ... */ },
  });
```

与 `workflow` 一起安装新包：

```bash
npm install @ai-sdk/workflow
```

### 写入 `ModelCallStreamPart`，而非 `UIMessageChunk`

`DurableAgent` 直接将 `UIMessageChunk` 对象写入 `getWritable()` 返回的 writable。`WorkflowAgent` 写入较低级别的 `ModelCallStreamPart` 形状，并在响应边界进行转换。这使持久流保持 provider 形状，避免将 UI 协议烘焙到 workflow 负载中。

```diff
  // Inside the workflow
  await agent.stream({
    messages,
-   writable: getWritable<UIMessageChunk>(),
+   writable: getWritable<ModelCallStreamPart>(),
  });
```

```diff
  // Inside the route handler
+ import { createModelCallToUIChunkTransform } from '@ai-sdk/workflow';

  return createUIMessageStreamResponse({
-   stream: run.readable,
+   stream: run.readable.pipeThrough(createModelCallToUIChunkTransform()),
  });
```

### 用 `stopWhen` 替换 `maxSteps`

`DurableAgent` 直接接受 `maxSteps`。`WorkflowAgent` 使用 AI SDK 的共享 `stopWhen` 条件，使相同的停止逻辑在 `ToolLoopAgent`、`generateText` 和 `streamText` 中生效。

```diff
+ import { isStepCount } from 'ai';

  await agent.stream({
    messages,
-   maxSteps: 10,
+   stopWhen: isStepCount(10),
  });
```

完整停止条件列表请参阅[循环控制](/docs/agents/loop-control)。

### 用 `output` 替换 `experimental_output`

```diff
+ import { Output } from '@ai-sdk/workflow';

  await agent.stream({
    messages,
-   experimental_output: Output.object({ schema }),
+   output: Output.object({ schema }),
  });
```

返回值现在在 `result.output` 上（先前为 `result.experimental_output`）。

### WorkflowAgent：对人在回路工具使用 `needsApproval`

使用 `DurableAgent` 时，工具审批通过在工具的 `execute` 函数内调用 Hook 实现。`WorkflowAgent` 将审批作为一流的工具属性——Agent 发出审批请求、挂起工作流，并在用户响应时自动恢复。

```diff
  bookFlight: tool({
    description: 'Book a flight',
    inputSchema: z.object({ flightId: z.string() }),
+   needsApproval: true,
-   execute: async (input) => {
-     const approved = await waitForApprovalHook(input);
-     if (!approved) throw new Error('Denied');
-     return bookFlightStep(input);
-   },
+   execute: bookFlightStep,
  }),
```

`needsApproval` 也接受异步函数，以便你可以按输入决定是否需要审批（见上文[工具审批](#tool-approval)）。

### `uiMessages` / `collectUIMessages` 已移除

当设置 `collectUIMessages: true` 时，`DurableAgent.stream()` 返回累积的 `uiMessages`。`WorkflowAgent.stream()` 在 `result.messages` 上返回 `ModelMessage[]`——持久化这些消息，并在边界使用 `convertToModelMessages` / `convertToUIMessages` 按需转换。

```diff
  const result = await agent.stream({
    messages,
    writable: getWritable<ModelCallStreamPart>(),
-   collectUIMessages: true,
  });

- return { uiMessages: result.uiMessages };
+ return { messages: result.messages };
```

### 没有 `generate()` 方法

`WorkflowAgent` 仅暴露 `stream()`。如果你曾调用 `agent.generate()`，请切换到 `stream()`，并在 promise 解析后读取 `result.messages` / `result.output`。

### 用 `runtimeContext` 和 `toolsContext` 替换 `experimental_context`

`WorkflowAgent` 不再接受 `experimental_context`。将值拆分为共享 Agent 状态（`runtimeContext`）和每工具状态（`toolsContext`）；每个工具的 `execute` 仅将其自己的已验证条目作为 `context` 接收。完整形状见 [runtimeContext 和 toolsContext](#runtimecontext-and-toolscontext)。

```diff
  const agent = new WorkflowAgent({
    model: 'anthropic/claude-sonnet-4-6',
    tools: { weather: weatherTool },
-   experimental_context: { tenantId: 'tenant_123', apiKey: 'sk-...' },
+   runtimeContext: { tenantId: 'tenant_123' },
+   toolsContext: { weather: { apiKey: 'sk-...' } },
  });
```

### 其他一切

其他选项以相同名称延续：`prepareStep`、`onStepFinish`、`onFinish`、`onError`、`toolChoice`、`activeTools`、`timeout`、`experimental_repairToolCall`，以及常用生成设置（`temperature`、`maxOutputTokens`、`topP` 等）。`WorkflowAgent` 此外还添加 `prepareCall`（在循环前运行一次）以及上文记录的生命周期回调 `experimental_onStart` / `experimental_onStepStart` / `onToolExecutionStart` / `onToolExecutionEnd`。
