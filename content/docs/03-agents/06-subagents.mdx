# 子 Agent

子 Agent 是父 Agent 可以调用的 Agent。父 Agent 通过工具委托工作，子 Agent 在返回结果前自主执行。

## 工作原理

1. **定义子 Agent**，包含其自己的模型、instructions 和 tools
2. **创建调用它的工具**，供主 Agent 使用
3. **子 Agent 在独立的上下文窗口中独立运行**
4. **返回结果**（可选地将进度流式传输到 UI）
5. **使用 `toModelOutput` 控制模型看到的内容**以进行摘要

## 何时使用子 Agent

子 Agent 会增加延迟和复杂度。在收益大于成本时使用：

| 使用子 Agent 当                              | 避免子 Agent 当           |
| ----------------------------------------------- | ------------------------------ |
| 任务需要探索大量 token | 任务简单且聚焦   |
| 需要并行化独立研究    | 顺序处理已足够 |
| 上下文会超出模型限制          | 上下文可管理       |
| 需要按能力隔离工具访问   | 所有工具可安全共存   |

## 为何使用子 Agent？

### 卸载上下文密集的任务

某些任务需要探索大量信息——读取文件、搜索代码库或研究主题。在主 Agent 中运行这些任务会快速消耗上下文，使 Agent 随时间变得不那么连贯。

使用子 Agent，你可以：

- 启动使用数十万 token 的专用 Agent
- 让它仅返回聚焦的摘要（也许 1,000 个 token）
- 保持主 Agent 的上下文清晰连贯

子 Agent 承担繁重工作，主 Agent 专注于编排。

### 并行化独立工作

对于探索代码库等任务，你可以同时启动多个子 Agent 研究不同区域。每个返回摘要，主 Agent 综合发现——而无需承担所有探索的上下文成本。

### 专用编排

一种较少见但有效的模式是使用主 Agent 纯粹进行编排，将不同类型的工作委托给专用子 Agent。例如：

- 带有只读工具用于研究代码库的探索子 Agent
- 带有文件编辑工具的编码子 Agent
- 带有特定平台或 API 工具的集成子 Agent

这创建了清晰的关注点分离，尽管上下文卸载和并行化是子 Agent 更常见的动机。

## 无流式的基础子 Agent

最简单的子 Agent 模式不需要特殊机制。主 Agent 有一个在其 `execute` 函数中调用另一个 Agent 的工具：

```ts
import { ToolLoopAgent, tool } from 'ai';
__PROVIDER_IMPORT__;
import { z } from 'zod';

// Define a subagent for research tasks
const researchSubagent = new ToolLoopAgent({
  model: __MODEL__,
  instructions: `You are a research agent.
Summarize your findings in your final response.`,
  tools: {
    read: readFileTool, // defined elsewhere
    search: searchTool, // defined elsewhere
  },
});

// Create a tool that delegates to the subagent
const researchTool = tool({
  description: 'Research a topic or question in depth.',
  inputSchema: z.object({
    task: z.string().describe('The research task to complete'),
  }),
  execute: async ({ task }, { abortSignal }) => {
    const result = await researchSubagent.generate({
      prompt: task,
      abortSignal,
    });
    return result.text;
  },
});

// Main agent uses the research tool
const mainAgent = new ToolLoopAgent({
  model: __MODEL__,
  instructions: 'You are a helpful assistant that can delegate research tasks.',
  tools: {
    research: researchTool,
  },
});
```

当你不需要在 UI 中显示子 Agent 的进度时，这很有效。工具调用会阻塞直到子 Agent 完成，然后返回最终文本响应。

### 处理取消

当用户取消请求时，`abortSignal` 会传播到子 Agent。始终传递它以确保清理：

```ts
execute: async ({ task }, { abortSignal }) => {
  const result = await researchSubagent.generate({
    prompt: task,
    abortSignal, // Cancels subagent if main request is aborted
  });
  return result.text;
},
```

如果你中止信号，子 Agent 停止执行并抛出 `AbortError`。主 Agent 的工具执行失败，这会停止主循环。

为避免后续消息中出现不完整工具调用的错误，请使用带有 `ignoreIncompleteToolCalls` 的 `convertToModelMessages`：

```ts
import { convertToModelMessages } from 'ai';

const modelMessages = await convertToModelMessages(messages, {
  ignoreIncompleteToolCalls: true,
});
```

这会过滤掉没有对应结果的工具调用。了解更多请参阅 [convertToModelMessages](/docs/reference/ai-sdk-ui/convert-to-model-messages) 参考。

## 流式传输子 Agent 进度

当你希望在子 Agent 工作时显示增量进度时，请使用[**初步工具结果**](/docs/ai-sdk-core/tools-and-tool-calling#preliminary-tool-results)。此模式使用生成器函数，向 UI 产生部分更新。

### 初步工具结果如何工作

将 `execute` 函数从常规函数改为异步生成器（`async function*`）。每次 `yield` 向前端发送初步结果：

```ts
execute: async function* ({ /* input */ }) {
  // ... do work ...
  yield partialResult;
  // ... do more work ...
  yield updatedResult;
}
```

### 构建完整消息

每次 `yield` **完全替换**先前的输出（不会追加）。这意味着你需要一种方式将子 Agent 的响应累积为随时间增长的完整消息。

`readUIMessageStream` 实用程序处理这一点。它读取流中的每个块，并构建包含迄今收到的所有部分的不断增长的 `UIMessage`：

```ts
import { readUIMessageStream, tool } from 'ai';
import { z } from 'zod';

const researchTool = tool({
  description: 'Research a topic or question in depth.',
  inputSchema: z.object({
    task: z.string().describe('The research task to complete'),
  }),
  execute: async function* ({ task }, { abortSignal }) {
    // Start the subagent with streaming
    const result = await researchSubagent.stream({
      prompt: task,
      abortSignal,
    });

    // Each iteration yields a complete, accumulated UIMessage
    for await (const message of readUIMessageStream({
      stream: result.toUIMessageStream(),
    })) {
      yield message;
    }
  },
});
```

每个产生的 `message` 是完整的 `UIMessage`，包含子 Agent 迄今的所有部分（文本、工具调用和工具结果）。前端只需用每条新消息替换其显示。

## 控制模型看到的内容

这正是子 Agent 在上下文管理方面变得强大的地方。包含子 Agent 全部工作的完整 `UIMessage` 存储在消息历史中并显示在 UI 中。但你可以使用 `toModelOutput` 控制主 Agent 的模型实际看到的内容。

### 工作原理

`toModelOutput` 函数将工具的输出映射为发送给模型的 token：

```ts
const researchTool = tool({
  description: 'Research a topic or question in depth.',
  inputSchema: z.object({
    task: z.string().describe('The research task to complete'),
  }),
  execute: async function* ({ task }, { abortSignal }) {
    const result = await researchSubagent.stream({
      prompt: task,
      abortSignal,
    });

    for await (const message of readUIMessageStream({
      stream: result.toUIMessageStream(),
    })) {
      yield message;
    }
  },
  toModelOutput: ({ output: message }) => {
    // Extract just the final text as a summary
    const lastTextPart = message?.parts.findLast(p => p.type === 'text');
    return {
      type: 'text',
      value: lastTextPart?.text ?? 'Task completed.',
    };
  },
});
```

使用此设置：

- **用户看到**：子 Agent 的完整执行——每个工具调用、每个中间步骤
- **模型看到**：仅最终摘要文本

子 Agent 可能使用 100,000 个 token 进行探索和推理，但主 Agent 仅消耗摘要。这使主 Agent 保持连贯和专注。

### 编写用于摘要的子 Agent 指令

为使 `toModelOutput` 提取有用的摘要，你的子 Agent 必须产生一个。添加类似以下的显式指令：

```ts
const researchSubagent = new ToolLoopAgent({
  model: __MODEL__,
  instructions: `You are a research agent. Complete the task autonomously.

IMPORTANT: When you have finished, write a clear summary of your findings as your final response.
This summary will be returned to the main agent, so include all relevant information.`,
  tools: {
    read: readFileTool,
    search: searchTool,
  },
});
```

没有此指令，子 Agent 可能不会产生全面的摘要。它可能只说 "Done"，使 `toModelOutput` 无法提取有用的内容。

## 在 UI 中渲染子 Agent（使用 useChat）

要显示流式进度，请检查工具部分的 `state` 和 `preliminary` 标志。

### 工具部分状态

| 状态              | 描述                                |
| ------------------ | ------------------------------------------ |
| `input-streaming`  | 正在生成工具输入                 |
| `input-available`  | 工具准备执行                      |
| `output-available` | 工具已产生输出（检查 `preliminary`） |
| `output-error`     | 工具执行失败                      |

### 检测流式 vs 完成

```tsx
const hasOutput = part.state === 'output-available';
const isStreaming = hasOutput && part.preliminary === true;
const isComplete = hasOutput && !part.preliminary;
```

### 子 Agent 输出的类型安全

在 UI 组件中导出与 Agent 一起使用的类型：

```ts filename="lib/agents.ts"
import { ToolLoopAgent, InferAgentUIMessage } from 'ai';

export const mainAgent = new ToolLoopAgent({
  // ... configuration with researchTool
});

// Export the main agent message type for the chat UI
export type MainAgentMessage = InferAgentUIMessage<typeof mainAgent>;
```

### 渲染消息和子 Agent 输出

此示例使用上面定义的类型渲染主 Agent 的消息和子 Agent 的流式输出：

```tsx
'use client';

import { useChat } from '@ai-sdk/react';
import type { MainAgentMessage } from '@/lib/agents';

export function Chat() {
  const { messages } = useChat<MainAgentMessage>();

  return (
    <div>
      {messages.map(message =>
        message.parts.map((part, i) => {
          switch (part.type) {
            case 'text':
              return <p key={i}>{part.text}</p>;
            case 'tool-research':
              return (
                <div>
                  {part.state !== 'input-streaming' && (
                    <div>Research: {part.input.task}</div>
                  )}
                  {part.state === 'output-available' && (
                    <div>
                      {part.output.parts.map((nestedPart, i) => {
                        switch (nestedPart.type) {
                          case 'text':
                            return <p key={i}>{nestedPart.text}</p>;
                          default:
                            return null;
                        }
                      })}
                    </div>
                  )}
                </div>
              );
            default:
              return null;
          }
        }),
      )}
    </div>
  );
}
```

## 注意事项

### 子 Agent 中不支持工具审批

子 Agent 工具不能使用 `toolApproval`（或已弃用的 `needsApproval`）等审批流程。所有工具必须自动执行，无需用户确认。

### 子 Agent 上下文是隔离的

每次子 Agent 调用都从零开始的全新上下文窗口开始。这是子 Agent 的关键优势之一：它们不会继承主 Agent 的累积上下文，这正是它们能够进行繁重探索而不膨胀主对话的原因。

如果你需要让子 Agent 访问对话历史，`messages` 在工具的 execute 函数中与 `abortSignal` 一起可用：

```ts
execute: async ({ task }, { abortSignal, messages }) => {
  const result = await researchSubagent.generate({
    messages: [
      ...messages, // The main agent's conversation history
      { role: 'user', content: task }, // The specific task for this invocation
    ],
    abortSignal,
  });
  return result.text;
},
```

请谨慎使用，因为传递完整历史会削弱部分上下文隔离的好处。

### 流式传输增加复杂度

基础模式（无流式）实现和调试更简单。仅当你需要在 UI 中显示实时进度时才添加流式传输。
