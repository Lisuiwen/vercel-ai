# 流背压与取消

本页重点介绍使用流时的背压和取消。你无需了解这些信息即可使用 AI SDK，但若感兴趣，可深入了解 SDK 为何以及如何最优地流式传输响应。

在以下章节中，我们将在简单示例程序的上下文中探讨背压和取消，讨论急切（eager）方法可能引发的问题，并演示惰性（lazy）方法如何解决它们。

## 流中的背压与取消

首先设置一个简单的示例程序：

```jsx
// A generator that will yield positive integers
async function* integers() {
  let i = 1;
  while (true) {
    console.log(`yielding ${i}`);
    yield i++;

    await sleep(100);
  }
}
function sleep(ms) {
  return new Promise(resolve => setTimeout(resolve, ms));
}

// Wraps a generator into a ReadableStream
function createStream(iterator) {
  return new ReadableStream({
    async start(controller) {
      for await (const v of iterator) {
        controller.enqueue(v);
      }
      controller.close();
    },
  });
}

// Collect data from stream
async function run() {
  // Set up a stream of integers
  const stream = createStream(integers());

  // Read values from our stream
  const reader = stream.getReader();
  for (let i = 0; i < 10_000; i++) {
    // we know our stream is infinite, so there's no need to check `done`.
    const { value } = await reader.read();
    console.log(`read ${value}`);

    await sleep(1_000);
  }
}
run();
```

在此示例中，我们创建了一个产出正整数的 async generator、包装该 generator 的 `ReadableStream`，以及从流中读取值的 reader。请注意，整数 generator 会输出 `"yielding ${i}"`，reader 会输出 `"read ${value}"`。两者处理数据都需要任意时间，generator 中用 100ms sleep 表示，reader 中用 1 秒 sleep 表示。

## 背压

若运行此程序，你会注意到有趣的现象：大约每有一条 "read" 日志就有 10 条 "yield" 日志。这可能看似显而易见 — generator 推送值的速度比 reader 拉取快 10 倍。但这代表一个问题：我们的 `stream` 必须维护一个不断扩大的队列，其中包含已推入但尚未拉出的项。

问题源于我们将 generator 包装成流的方式。注意 `start` 处理程序中 `for await (…)` 的使用。这是一个**急切**的 for 循环，它不断运行以从 generator 获取下一个值并入队到流中。这意味着流不尊重背压 — 即消费者向生产者发出的「暂时不需要更多值」的信号。我们实质上启动了一个会持续向流推送数据的线程，并尽可能快地立即推送新数据。更糟的是，当我们不需要额外数据时，无法向该线程发出停止信号。

要修复此问题，`ReadableStream` 提供 `pull` 处理程序。每当消费者尝试从流读取更多数据时（若内部尚未排队数据），就会调用 `pull`。但仅将 `for await(…)` 移入 `pull` 还不够，还需要从急切入队转为**惰性**入队。通过这两处修改，我们就能响应消费者：若需要更多数据，可轻松生产；若不需要，则不必做不必要的工作。

```jsx
function createStream(iterator) {
  return new ReadableStream({
    async pull(controller) {
      const { value, done } = await iterator.next();

      if (done) {
        controller.close();
      } else {
        controller.enqueue(value);
      }
    },
  });
}
```

我们的 `createStream` 现在稍冗长，但新代码很重要。首先，需要手动调用 `iterator.next()`。它返回类型为 `{ done: boolean, value: T }` 的对象的 `Promise`。若 `done` 为 `true`，则 iterator 不会再产出值，必须 `close` 流（让消费者知道流也已完成产出）。否则，需要 `enqueue` 新产生的值。

运行此程序时，会看到 "yield" 和 "read" 日志成对出现。我们不再为每次 read 产出 10 倍整数！并且，流现在只需在内部缓冲区维护 1 个项。我们实质上把控制权交给了消费者，由它按需负责生产新值。很棒！

## 取消

回到最初的急切示例，做一个小修改。现在只读取 3 个整数，而不是 10,000 个：

```jsx
// A generator that will yield positive integers
async function* integers() {
  let i = 1;
  while (true) {
    console.log(`yielding ${i}`);
    yield i++;

    await sleep(100);
  }
}
function sleep(ms) {
  return new Promise(resolve => setTimeout(resolve, ms));
}

// Wraps a generator into a ReadableStream
function createStream(iterator) {
  return new ReadableStream({
    async start(controller) {
      for await (const v of iterator) {
        controller.enqueue(v);
      }
      controller.close();
    },
  });
}
// Collect data from stream
async function run() {
  // Set up a stream that of integers
  const stream = createStream(integers());

  // Read values from our stream
  const reader = stream.getReader();
  // We're only reading 3 items this time:
  for (let i = 0; i < 3; i++) {
    // we know our stream is infinite, so there's no need to check `done`.
    const { value } = await reader.read();
    console.log(`read ${value}`);

    await sleep(1000);
  }
}
run();
```

我们又回到读出的值数量约 10 倍的 yield。但注意，读取 3 个值后，仍在继续 yield 新值。我们知道 reader 不会再读取，但流不知道！急切的 `for await (…)` 会永远继续，不断将新值入队到流缓冲区，增加内存使用，直到耗尽所有可用程序内存。

修复方法完全相同：使用 `pull` 和手动迭代。通过**惰性**生产值，我们将整数 generator 的生命周期与 reader 绑定。一旦停止读取，yield 也会停止：

```jsx
// Wraps a generator into a ReadableStream
function createStream(iterator) {
  return new ReadableStream({
    async pull(controller) {
      const { value, done } = await iterator.next();

      if (done) {
        controller.close();
      } else {
        controller.enqueue(value);
      }
    },
  });
}
```

由于解决方案与实现背压相同，说明它们只是同一问题的两个侧面：向流推送值应**惰性**进行，急切推送会导致预期的问题。

## 将流惰性与 AI 响应关联

假设你正在将 AIBot 服务集成到产品中。用户可以提示「从 1 数到无穷」，浏览器会请求你的 AI API 端点，服务器连接 AIBot 获取响应。但「无穷」就是无穷的，响应永远不会结束！

几秒后用户感到无聊并离开页面。或者你在本地开发，热模块重载刷新了页面。浏览器已结束与 API 端点的连接，但服务器会结束与 AIBot 的连接吗？

若使用急切的 `for await (...)` 方法，连接仍在运行，服务器不断向 AIBot 请求更多数据。服务器启动了「线程」，且无法在需要时结束急切拉取。最终服务器将耗尽内存（记住，没有活跃的 fetch 连接来读取缓冲响应并释放它们）。

使用惰性方法，这些会为你处理好。因为流仅在消费者请求时才向 AIBot 请求新数据，离开页面会自然释放所有资源。fetch 连接中止，服务器可清理响应。与该响应绑定的 `ReadableStream` 可被垃圾回收。此时，它持有的与 AIBot 的连接也可被释放。
