# 中止与可恢复流

## 问题

在 `useChat` 中使用 `resume: true` 做流恢复时，客户端中止会被视为断开连接。关闭标签页、刷新、离开页面或调用 `stop()` 会关闭当前 HTTP 连接，但不应取消底层生成任务。

若应用把请求的 abort signal 一直传到模型调用，断开可能会取消「流恢复」期望继续运行的工作。若停止按钮只调用 `stop()`，服务端生成可能仍在继续，客户端可能重连到同一条活动流。

```tsx
const { messages, stop } = useChat({
  id: chatId,
  resume: true, // Stream resumption enabled
});

// stop() only aborts the current client request.
// It is not a server-side cancellation request.
```

## 背景

流恢复允许客户端在原始连接关闭后重连到仍在进行的活动流。为此，即使没有客户端当前连接，服务端也需要让流的生产端继续运行。

这意味着路由清理、页面卸载与网络断开应作为可恢复的断开处理。明确的用户取消需要单独的服务端信号，用于取消活动生产者并清除已保存的活动流引用。

## 解决方案

使用 `resume: true` 处理断开后的重连，并单独提供用于明确取消的 stop 接口。

stop 接口应：

1. 加载会话并读取其活动流 ID
2. 若客户端传入最新的部分助理消息，则持久化
3. 取消正在生成该流的工作
4. 仅当活动流引用仍指向同一条流时再清除它

在客户端，在停止本地聊天流之前先调用 stop 接口：

```tsx
const chat = useChat({
  id: chatId,
  resume: true,
});

async function stopStream() {
  await fetch(`/api/chat/${chatId}/stop`, { method: 'POST' });
  chat.stop();
}
```

将导航行为与停止行为分开。不要在路由清理、页面 `unload` 或组件卸载清理里调用 stop 接口——那些属于应保持可恢复的断开。

若不需要流恢复、只需客户端取消，请关闭 `resume` 并直接使用 `stop()`：

```tsx
const { messages, sendMessage, stop } = useChat({
  id: chatId,
  resume: false, // Disable stream resumption (default behavior)
});
```
